# ADR-0004:冒險手記 MapView 嚴格走過版 + NarrativePage 共用版型

- 狀態:Accepted
- 日期:2026-05-17
- 決策者:haloflag01

## 背景

冒險手記 UI(`scripts/ui/adventure_journal.gd`)Stage 1+2 完成後,已有:
- 三層 view:WorldView ⇄ MapView ⇄ PageView
- PageView 雙聲道校準對話精確版(對應戰鬥紀錄系統設計 §7.1)

但 MapView 目前是純橫排條列(連戰列 + 重挑橫列),沒有「玩家行動製造地圖」的視覺敘事感(設計 §一),用戶要求重做。

依用戶示意圖,新 MapView 結構:
- **垂直主軸**串聯主節點(戰役開始敘事 / 連戰 / 整備 / 結局敘事)
- **水平分支**從連戰主節點向右展開,同列 = 一次嘗試,多列 = 多次重挑(old/new)
- **X 標記** = GAME OVER;**方框 + 箭頭** = 菁英化 / clone spawn 連到後續節點

同時用戶提案新增 **NarrativePage 共用版型**(左 = 敘事文字 / 右上 = 場景插圖 / 右下 = 獲得區),把整備頁併入此版型,未來新節點類型(序章後半敘事 / Hub 對話 / 主冒險章節)直接套用。

設計分支多,跑 `/grill-with-docs` 走 7 個關鍵決定。

## 決定

### 1. MapView 主軸:嚴格「走過版」

**只畫玩家實際走過的主節點**,不預畫整個 11 節點骨架。

| 主節點類型 | 「走過」判定 |
|---|---|
| 戰役開始敘事 | 有 chain_index=0 的 BattleRecord |
| 連戰 N | 有 chain_index=N-1 的 BattleRecord |
| 整備 N | 有 chain_index_completed=N-1 的 PrepNodeRecord |
| 結局敘事 | `GameState.is_campaign_complete(campaign_id)` |
| 全空 | EmptyHint「(此戰役尚無紀錄)」 |

**為什麼選嚴格版**:對齊「玩家的行動製造了地圖」哲學(設計 §一)。「預畫結構」會讓 MapView 變成「campaign spec 預覽」,失去個人化敘事重量。spoiler 風險也低 — 第一期序章結構在 docs 公開,沒祕密可保留。

### 2. 主軸節點點擊互動

- **連戰主節點不可點**(純結構,僅作為視覺主軸節點)
- **敘事節點可點** → 開 NarrativePage(戰役開始 / 結局)
- **整備節點可點** → 開 NarrativePage(因併入共用版型)
- **戰鬥分支節點可點** → 開 PageView 戰鬥頁(現有)

**為什麼連戰主節點不可點**:它代表「這個連戰你打過」這個事實,具體戰鬥內容已透過分支節點承載;點連戰主節點要嘛跳第一場(快捷功能,價值不高)、要嘛開「連戰摘要頁」(新頁面類型,設計沒有)。**省下實作就讓它純結構**。

### 3. 戰鬥分支 old/new 視覺

- **整列透明度**(old 列 0.5 半透 / new 列 1.0 飽和)
- **列尾文字標記** old/new 輔助

new 列 = 該 chain 最大 retry_count 的列;其他都是 old。

**為什麼用整列透明度**:一眼能識別「最終那次是哪一列」,跟「過去的嘗試只是歷史」。比個別節點顏色組合(灰節點 + 紅 X 等)清爽,不會跟其他視覺編碼衝突。

### 4. 戰鬥節點失敗類型視覺對應

| failure_outcome | 視覺 | 連線 |
|---|---|---|
| -1(OHK)| 紅實心圓 | 連到下一節點 |
| `GRAY_RABBIT_CLONE_SPAWN` | 紅圓 + **黑方框** | 箭頭到 next position(clone 在 queue[0])|
| `WOLF_ELITE_PROMOTION` | 紅圓 + **黑方框** | 箭頭到該列末段的 is_elite=true 同 instance 節點 |
| `FOX_FLEE` | **暗灰圓** | 連到下一節點 |
| `RABBIT_CHAIN_FLEE` | 暗灰圓 × N | 同上(目前 dead branch) |
| `GAME_OVER` | **大 X 標記** | **該列終止,不連到下一節點** |

**為什麼 FOX_FLEE 用暗灰圓**:跟「OHK 紅」「GAME OVER X」三向區隔,中性表達「沒打死也沒終結」。
**為什麼 GAME OVER 是 X 而非節點 + 標記**:該列就此終止,需要明顯的「終結」記號;X 大標記跟其他節點視覺差異最大。
**為什麼 SPAWN/PROMOTION 都用方框**:它們在機制上同類型(原敵人離場 + 衍生新節點到 queue),視覺一致;箭頭表達「後續會再見到」。

### 5. NarrativePage 共用版型(整備頁併入)

新版型:
```
┌──────────────────────────┬──────────────────────────┐
│ 左頁:敘事/對話文字        │ 右上:場景插圖              │
│  + 時間/metadata(底)     │ ─── HSeparator ───        │
│                          │ 右下:獲得/變化區           │
└──────────────────────────┴──────────────────────────┘
```

**所有 narrative-like 節點共用此版型**:整備節點、戰役開始敘事、結局敘事(未來:序章後半敘事 / Hub 對話 / 主冒險章節)。差別只在「右下放什麼」。

**為什麼併入**:
1. 一個 render 函式統一邏輯;未來加新節點類型不必各寫一套
2. 對齊設計 §4.2 整備節點頁規範「左頁手寫風劇情 + 右頁地點美術圖」
3. 色塊 pass 的「場景區 placeholder」不醜(跟敵人色塊同手法);美術 pass 後所有節點同步升級

### 6. 戰役開始敘事右下「獲得區」

**起手卡組**(`campaign.starting_deck` → SupplyChip × N)。

**為什麼**:跟整備頁右下完全同語意(都是資源獲取),render 程式碼可直接重用。戰役開始本來就是「裝備好的時刻」,玩家在後續看到「我一開始拿到什麼 vs 現在剩什麼」,可形成對比。

### 7. 結局敘事右下「獲得區」

**解鎖的紀念品**(`GameState.unlocked_souvenirs` ∩ `SOUVENIR_INFO`)。

**為什麼**:對應設計 §7.4「紀念品 = 手記中夾頁物件」概念。結局頁是紀念品最自然的呈現點 — 「這場冒險留下了什麼」。跟 Hub 紀念品擺設共用同一個 `SOUVENIR_INFO` 字典(目前 hardcoded 在 `hub.gd`,未來提取為 .tres)。

### 8. 敘事節點不新增 record 類型

敘事節點(戰役開始 / 結局)**從 `CampaignDefinition` 衍生**,**不新增 NarrativeRecord**:
- 敘事文字 = `campaign_def.prologue_narrative` / `ending_narrative`
- 不寫進 `AdventureRecord`(沒有「玩家行動產生此節點」的真實事件)
- MapView 顯示與否,依「相關 BattleRecord / GameState 存在」推導

**為什麼**:敘事是 campaign 結構的一部分,不是玩家行動痕跡。寫 record 等於把固定結構複製到玩家存檔,沒新增資訊。

## 後果

正面:

- MapView 變成有結構的地圖視覺,符合設計 §一「玩家行動製造地圖」哲學
- NarrativePage 共用版型 → 未來節點類型無痛擴展(序章後半 / Hub 對話 / 主冒險章節)
- 整備頁併入後 render 邏輯統一,維護成本降
- 失敗類型有專屬視覺(X / 框 / 暗灰),失敗痕跡有敘事感(設計 §一第 3 個 use case)
- old/new 透明度 + 文字標記 → 多重挑連戰一眼看出進展

負面 / 取捨:

- MapView 重寫,layout engine 從「橫排 grid」變「主軸 + 分支 + 連線/箭頭」,複雜度 5x
- 敘事節點從 CampaignDefinition 衍生 → 跨戰役寫共用 campaign 結構解析邏輯(目前只 1 個 campaign 不痛,未來戰役多時要小心)
- 嚴格走過版 → 玩家看不到「未來戰役有多長」(設計 §3.1 提到「空心圈未走戰役」,我們選擇不畫)
- 連戰主節點不可點 → 玩家不能從主軸快速跳轉到該連戰的戰鬥;必須點分支節點(設計可接受)
- 結局右下 SOUVENIR_INFO 目前 hardcoded 在 hub.gd → 紀念品多時要提取 .tres(已知技術債)

## 已考慮的替代方案

### MapView 主軸:預畫所有主節點 + 未走顯示空心圈

- 拒絕原因:設計 §3.1「空心圈未走戰役」原本是針對 WorldView 跨戰役地圖(spoiler-free 開放世界感),不是 MapView 內單戰役的逐連戰結構。MapView 用預畫等於「給玩家看 spec」,違反 §一「行動製造地圖」哲學
- 若未來戰役有「隱藏連戰」(達成某條件才解鎖)要 spoiler 暗示,再回頭評估混合策略

### 連戰主節點可點 → 跳該連戰 new 列首場戰鬥

- 拒絕原因:快捷功能,但分支節點本身已可點,差別僅在「跳第一場」vs「自己選哪場」。為一個微小快捷加 click handler 不值
- 若未來戰役連戰場次極多(>10),這個快捷會有實質價值,屆時加

### 戰鬥節點 old/new 用個別節點顏色區分(灰節點 / 紅 ✓ 等)

- 拒絕原因:跟其他編碼(失敗 X / 方框 / 暗灰 FOX_FLEE)疊加會視覺爆炸,玩家難分辨「灰是 old 還是 FOX_FLEE」
- 整列透明度 = 單一維度區分,跟節點內部視覺正交,認知負擔低

### 整備頁保留現有獨立版型 + 敘事頁另寫一個 render 函式

- 拒絕原因:整備頁跟敘事頁 80% 重複邏輯;短期省事但長期 drift(改其中一個忘了同步另一個),且未來加新節點類型(序章後半 / 主冒險章節)時要寫第三套
- 共用版型一次性投入,長期維護成本最低

### 敘事節點新增 NarrativeRecord 類型

- 拒絕原因:敘事文字本就在 CampaignDefinition,記在玩家存檔等於 100% 重複資料;且擴 AdventureRecord schema 要 VERSION bump + migration(ADR-0003 機制),代價遠超回報
- 若未來「對話節點」需要記玩家選了哪個分支對話(state-bearing),才會需要 record 類型;敘事節點不需要

### 結局右下放戰役完成統計(總戰鬥數 / OHK 數 / 重挑數)

- 拒絕原因:「資料庫式統計」跟「冒險手記式敘事重量」哲學衝突(設計 §一表)。紀念品 = 物件,有具體性 + 紀念性;統計 = 抽象數字
- 若未來真有「成就系統」需求,獨立做成就頁,不擠進結局
