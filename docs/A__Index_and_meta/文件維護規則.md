# P6 文件維護規則

> **狀態**: 規範
> **適用範圍**: 所有 `docs/` 下與根目錄 `CONTEXT.md` 之外的活躍 .md 文件
> **最後同步**: 2026-05-16

文件分類軸 + banner 格式 + 何時 / 由誰更新狀態的單一規範。
本檔本身是「規範」類,沒有「實作完成」概念,新增分類時擴章節。

---

## 一、分類軸(5 主軸 + 附加群)

### 主軸 5 類(任一活躍 doc 必歸其一)

每類**物理對應一個子目錄**(2026-05-16 起):

| 類 | 物理位置 | 定義 | 銜接慣例 |
|---|---|---|---|
| **設計規範** | `docs/B__Design_specifications/` | 永久 reference / rule book,新增內容(卡 / 敵人 / UI 元件)時擴章節 | 動工新增 X 前必讀對應規範 |
| **實作基準** | `docs/C__Implementation_benchmarks/` | 設計已收斂 + code 已對應 + 本期內不大改 | 動 code 對齊此處規格 |
| **待實作 / 工作中** | `docs/D__In_progress/` | 規格寫了 / 部分寫了 / 沒進 code | 等里程碑進度 |
| **後續期數待決策** | `docs/E__Pending_decision/` | 第一期不做 | 第一期 scope 不納入 |
| **歷史 / 過時** | `docs/F__History/` | 設計被取代,留作演化記錄 | 不再為設計依據,只供考古 |

### 附加群(獨立,不在主軸)

| 群 | 物理位置 | 定義 | 範圍 |
|---|---|---|---|
| **參考資料** | `docs/A__Index_and_meta/參考作品分析.md` | 不是 P6 設計,是外部對標資料 | 1 份 |
| **meta / 工具** | `docs/A__Index_and_meta/`(其他)+ `/CONTEXT.md` | 工程工作流、領域語言摘要、ADR、索引 | CONTEXT.md / agents/* / adr/* / skills 指南 / 文件目錄 / 本檔 |

## 二、Banner 格式範本

每份「活躍」doc 頂部(緊接 `# 標題` 之後)放 banner。
歷史過時類(`docs/F__History/` 內)放精簡版即可。
meta / 工具群不要求 banner(自說明)。

### 設計規範

```markdown
# <文件標題>

> **狀態**: 設計規範
> **適用範圍**: <用在哪些情境 / 給誰參考>
> **最後同步**: YYYY-MM-DD
```

### 實作基準

```markdown
# <文件標題>

> **狀態**: 實作基準
> **對應實作**: <code 在哪 / 哪個 milestone 完成>
> **未實作條目**: <列點;若無寫「無」>
> **最後同步**: YYYY-MM-DD
```

### 待實作 / 工作中

```markdown
# <文件標題>

> **狀態**: 待實作 / 工作中
> **規格進度**: <已寫 / 待寫部分>
> **何時實作**: <里程碑 / 未定>
> **最後同步**: YYYY-MM-DD
```

### 後續期數待決策

```markdown
# <文件標題>

> **狀態**: 後續期數待決策
> **第一期是否實作**: <否 / 部分(說明)>
> **待決策問題**: <列點>
> **最後同步**: YYYY-MM-DD
```

### 歷史 / 過時(精簡)

```markdown
# <文件標題>

> **狀態**: 歷史 / 已被取代
> **被誰取代**: <當前主線文件名 + 章節>
```

### 參考資料

```markdown
# <文件標題>

> **狀態**: 參考資料
> **性質**: 對標 / 外部資料,非 P6 設計
> **最後同步**: YYYY-MM-DD
```

## 三、何時更新 banner(三條核心規則)

### 規則 1:動 code 影響某 doc 的實作狀態 → 同個 commit 內更新該 doc banner

**情境**:
- 動 `scripts/ui/m2_campaign.gd` 加新 Phase → 看是否影響 `遊戲核心系統機制.md` 的 Phase 列表;若是 → 同個 commit 改該 doc banner「未實作條目」與「最後同步」
- 實作 `戰鬥紀錄系統設計.md` §6 跨期整合 → 該 doc banner 移除 §6 從「未實作條目」+ 更新「最後同步」

**反例**:
- 純 refactor(不改外部行為)→ doc 不動 banner
- 純 bug fix → doc 不動 banner(除非 doc 本身寫錯)

### 規則 2:完成 milestone(M5-S2 / M6 等)→ bump 對應 doc + `文件目錄.md` 索引

完成一個 milestone 時必做:
- 對應 doc banner 加「對應實作」欄(若是首次實作)或更新該欄(新增 milestone)
- `docs/A__Index_and_meta/文件目錄.md` 的「最後同步」欄反映此 milestone
- 該 doc 若狀態 / 類別變了(例如從「待實作」→「實作基準」),banner 整段更新 **+ 物理 git mv 到對應子目錄**(B__ / C__ / D__ / E__ / F__)+ 跑 `tools/rewrite_doc_links.py`(或手動更新引用)

### 規則 3:加新 doc → 必須附 banner + 放對子目錄 + 加進 `文件目錄.md` 對應區塊

**新 doc checklist**:
1. 決定屬哪一類(設計規範 / 實作基準 / 待實作 / 後續期數待決策),放對應子目錄
2. 頂部加 banner(按本檔 §二 範本)
3. 在 `docs/A__Index_and_meta/文件目錄.md` 對應區塊加一行
4. 若新 doc 對應某段 code → 該 code 加註標明出處(可選)

**禁止**:新增 doc 卻不歸類 / 不加 banner / 不進索引。寧可標「待決策」也不能無類。

## 四、誰負責

P6 是個人專案 + AI 協作。維護責任分工:

| 角色 | 何時觸發更新 |
|---|---|
| **使用者(人類)** | 完成 milestone 時、決定文件分類變化時、新增 doc 時 |
| **AI(Claude Code)** | 每 session 自動讀 `CLAUDE.md` → 進到本檔 → 動 code 時主動 check 三條規則 |

CLAUDE.md 已加 pointer 指向本檔。AI 動 docs/ 或對應 code 時應先 check 本檔。
