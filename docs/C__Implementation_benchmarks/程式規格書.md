## 一、平台與技術棧

### 目標平台

| 平台 | 優先級 | 備註 |
|------|--------|------|
| Windows | 主要 | Steam / itch.io 主力 |
| macOS | 次要 | Godot 跨平台輸出附帶 |
| Linux | 次要 | 同上 |
| 行動端 | **不在第一期** | 後續可能擴充 |

### 最低硬體建議

- CPU:雙核 2.0 GHz 以上
- RAM:2 GB
- GPU:支援 OpenGL 3.3 / Vulkan 1.0(整合顯卡即可)
- 儲存空間:500 MB(第一期內容)
- 解析度支援:1280×720 起跳,**1080p 為設計基準**

### 引擎與語言

**主引擎:Godot 4.x**(建議使用 4.2+ 穩定版)

**主語言:GDScript**——P6 邏輯複雜度可控,GDScript 完全足夠。

**備用:C#**——僅在以下情境考慮:
- 大量數值計算的效能瓶頸(例如:本擊上限討論若採動態算法)
- 需要重用既有 .NET 套件
- 開發者偏好

### 為什麼選 Godot

| 理由 | 說明 |
|------|------|
| **免費開源** | 無授權成本,適合個人開發者長期維護 |
| **AI 訓練資源足夠** | GDScript / Godot 4.x 在 AI 模型訓練資料中覆蓋良好,適合 AI 輔助寫 code |
| **2D 戰術遊戲對位精準** | Godot 的 2D 支援是一等公民(不像 Unity 的 2D 是次要) |
| **個人開發者友善** | 啟動快、編輯器輕量、無付費綁定 |
| **跨平台輸出簡單** | Windows/Mac/Linux 一鍵輸出 |
| **場景組合優於繼承** | 適合 P6 的「模組組合」設計風格 |

### 為什麼不選其他

- **Unity**:授權變動風險、相對重、3D 為主導向(我們是純 2D)
- **Unreal**:過度為大型 3D 設計,個人 2D 開發者大材小用
- **Web 技術**:平台選 PC,無需 Web 框架的限制

## 二、系統架構總覽

### 分層架構

```
┌─────────────────────────────────────┐
│  呈現層 (Presentation)              │
│  ── UI 元件、動畫、音效              │
├─────────────────────────────────────┤
│  邏輯層 (Logic / Game Rules)        │
│  ── 戰鬥引擎、戰役狀態機、失敗處理    │
├─────────────────────────────────────┤
│  資料層 (Data Models)               │
│  ── 卡牌、敵人、戰役、對話、紀念品    │
├─────────────────────────────────────┤
│  持久化層 (Persistence)             │
│  ── 存檔、戰鬥紀錄、設置             │
└─────────────────────────────────────┘
```

**依賴方向**:**上層依賴下層,下層不知上層**。
- UI 讀資料、呼叫邏輯,但邏輯不直接操作 UI(透過 Signal)
- 邏輯讀資料,但資料不知道邏輯
- 持久化獨立於前三層,以 API 提供存取

### 模組依賴關係

```
[UI 層] ←─── Signal ─── [戰鬥引擎] ←──── [卡牌資料層]
   ↑                        ↑               ↑
   └── Signal ──── [戰役狀態機] ────────── [敵人資料層]
                       ↑                     ↑
                       ├── [失敗代價系統]
                       ├── [失敗重啟機制]
                       ├── [整備節點]
                       ├── [Hub 管理]
                       └── [對話系統]
                              ↓
                       [存檔 / 戰鬥紀錄]
```

### Autoload(全域單例)

實際 `project.godot` 註冊的 autoload:

- **EventBus**:跨模組 Signal 轉發中心
- **CampaignContext**:當前戰役上下文(含 `tutorial_retry_enabled` / `retry_count` / `deck_snapshot`)
- **SaveSystem**:存檔讀寫(prep node auto-save)
- **GameState**:當前戰役、玩家進度、紀念品解鎖
- **AdventureRecord**:冒險手記紀錄(M5-S1;戰鬥 + 整備節點 append-only log)
- **ResourceLibrary**:靜態資源目錄 — 所有 .tres(卡牌 / 敵人 / 連戰 / 整備 / 戰役 / 校準台詞)的唯一入口

預留未實作:
- **Localization**:多語言(第一期可單語,但預留)

#### ResourceLibrary 規則(2026-05-16 起)

**新增任何靜態 .tres 資源 = 必須在 `scripts/autoload/resource_library.gd` 註冊**:

1. 把 .tres 放到 `res://resources/<type>/` 對應子目錄
2. 在 `resource_library.gd` 頂部對應的 `*_PATHS` dict 加一行 `id → 路徑`
3. **不可**在其他場景使用 `load("res://resources/...")` 載入這類資源;一律透過 `ResourceLibrary.<type>(id)` 取得

對外介面(typed methods):

```gdscript
ResourceLibrary.card(id) -> CardDefinition
ResourceLibrary.enemy_template(id) -> EnemyTemplate
ResourceLibrary.enemy_instance(id) -> EnemyInstance
ResourceLibrary.chain(id) -> ChainDefinition
ResourceLibrary.supply(id) -> SupplyPhase
ResourceLibrary.campaign(id) -> CampaignDefinition
ResourceLibrary.calibration_lines() -> CalibrationLines

# 批量(遍歷用)
ResourceLibrary.cards() / enemy_templates() / enemy_instances()
ResourceLibrary.chains() / supplies() / campaigns()
```

失敗模式:id 找不到 → `push_error` + 回 `null`。caller 自己負責容錯。

設計決策見 [docs/adr/0001-resource-library-autoload.md](../A__Index_and_meta/adr/0001-resource-library-autoload.md)。

## 三、核心模組規格

### 3.1 卡牌資料層

**職責**:管理卡牌靜態定義 + 戰役內卡組動態狀態。

> **2026-05-14 改版**:移除 `use_limit` / `pool_size`,改用「卡組張數」表達(一張卡 = 一次使用)。

**資料格式**:Godot Resource(.tres)為主。每張卡是一個 `CardDefinition` Resource。

```gdscript
# scripts/data/card_definition.gd
class_name CardDefinition
extends Resource

@export var id: String                    # "tool_arrow_pierce"
@export var card_name: String             # "射箭"
@export var resource_class: String        # "tool" / "burst" / "character"
@export var function_class: String        # "combat" / "intel" / "compound"
@export var card_form: String             # "pool" / "individual"
@export var weakness_type: String         # "impact" / "pierce" / "burn" / "generic" / "none"
@export var contribution: Dictionary      # { "pierce": 1 } 或 { "flexible": 2 } 或 {}
@export var strength_level: int           # 1 / 2 / 3
@export var lock_class: String            # "none" / "optional" / "required"
@export var special_effect: Resource      # SpecialEffect 子資源
@export var description: String
```

**動態狀態(戰役運行時)**:卡組以「卡定義 × 該類張數」清單管理。

```gdscript
# scripts/data/deck_entry.gd
class_name DeckEntry
extends Resource

@export var card_id: String
@export var count_remaining: int          # 卡組中該類卡剩餘張數
@export var count_consumed_total: int     # 戰役期間累積消耗(供戰鬥紀錄使用)
```

序章前半起手卡組範例:
```
[
  { card_id: "tool_arrow_pierce",  count_remaining: 10, count_consumed_total: 0 },
  { card_id: "tool_stone_impact",  count_remaining: 8,  count_consumed_total: 0 },
  { card_id: "intel_weakness",     count_remaining: 2,  count_consumed_total: 0 }
]
```

**模組對外介面**:
- `CardLibrary.load_all() -> Array[CardDefinition]`
- `CardLibrary.get_by_id(id: String) -> CardDefinition`
- `DeckManager.consume(card_id, amount=1)` — 對應 count_remaining -= amount
- `DeckManager.add(card_id, amount)` — 對應整備補給
- `DeckManager.snapshot() -> DeckSnapshot` — 為「教學重來」使用
- `DeckManager.restore(snapshot)` — 教學重來時回滾

**參考**:[卡牌設計原則.md](../B__Design_specifications/卡牌設計原則.md) §7 + [卡牌資料庫.md](卡牌資料庫.md)

**📚 舊專案參考**:`old_game_plan/core/Models/ActionCard.cs` 提供 C# Record 資料設計的結構參考。詳見 §9 速查表。

### 3.2 敵人資料層

**職責**:管理 5 模板 + instance 變異。

**資料格式**:`EnemyTemplate` Resource + `EnemyInstance` 子 Resource。

```gdscript
# scripts/data/enemy_template.gd
class_name EnemyTemplate
extends Resource

@export var id: String
@export var template_name: String
@export var enemy_class: String           # "rabbit" / "wolf" / "leopard" / "bear"
@export var weakness_range: Array[String] # 弱點範圍
@export var combat_state_distribution: Dictionary  # 戰鬥狀態傾向機率
@export var instances: Array[EnemyInstance]        # 該模板的 instance 列表
@export var design_contract: Dictionary   # 教 / 老手能做到 / 新手能理解

# 後續期數預留 — 第一期空陣列
@export var traits: Array = []            # 敵人特性(主動行為);詳見 敵人主動行為提案.md
```

**⭐ `traits` 欄位**(2026-05-13 加入,預留):
- **第一期所有敵人 traits 都是空陣列**
- 後續期數加入特性(反擊型 / 威脅型 / 傳染型)時填充
- 詳見 [敵人主動行為提案.md](../E__Pending_decision/敵人主動行為提案.md)

```gdscript
# scripts/data/enemy_instance.gd
class_name EnemyInstance
extends Resource

@export var instance_id: String           # "sentinel_weak_impact_low"
@export var requirements: Dictionary      # { "impact": 1, "pierce": 4, "mixed": 6 }
                                          # mixed = max(other types) × 1.5(2026-05-13 定案)
@export var strike_limit: int             # 本擊上限(2026-05-13 加入)
                                          # 計算規則見 敵人設計原則.md §6.2
@export var visual_cues: Dictionary       # 對應視覺線索詞彙表
```

**⭐ `requirements.mixed`**(2026-05-13 公式定案):
- 計算公式:**該敵人最高類型需求 × 1.5,取整**
- 範例:衝擊 4 / 穿刺 7 / 燃燒 7 → mixed = 7 × 1.5 = 11

**⭐ `strike_limit`**(2026-05-13 加入):
- 本擊上限,玩家本擊不能超過此張數
- 計算規則(依敵人類別):
  - 普通(兔/狼)= 最低需求 × 2
  - 精英狼 / 豹 = 最低需求 × 2.5
  - 熊(BOSS)= 最低需求 × 3 + 5
- 環境影響後續加入

**模組對外介面**:
- `EnemyLibrary.load_all() -> Array[EnemyTemplate]`
- `EnemyLibrary.pick_instance(template_id, rules) -> EnemyInstance`
- `EnemyState.create_from_instance(instance) -> EnemyState`

**參考**:[敵人設計原則.md](../B__Design_specifications/敵人設計原則.md) + [敵人資料庫.md](敵人資料庫.md)

### 3.3 戰鬥引擎

### 3.4 失敗代價系統

**職責**:依敵人類別處理失敗後果。**教學重來機制**優先檢查。

```gdscript
# scripts/logic/failure_handler.gd
class_name FailureHandler

func handle_failure(enemy: EnemyState, strike: Strike, ctx: CampaignContext):
    # 優先檢查教學重來(僅序章前半第一連戰)
    if ctx.tutorial_retry_enabled and enemy.template.enemy_class == "rabbit":
        TutorialRetry.on_rabbit_failed(ctx)
        return  # 早退;不進入正式失敗代價分支

    match enemy.template.enemy_class:
        "rabbit":  _handle_rabbit_failure(enemy, strike, ctx)
        "wolf":    _handle_wolf_failure(enemy, strike, ctx)
        "leopard", "bear": _trigger_game_over(ctx)
```

**核心邏輯**:
- 兔子型(序章前半):任一兔子失敗 → **連鎖逃跑**(連戰中其他兔子也逃走)→ 連戰結束
- 兔子型(序章後半 / 後續期數):從連戰移除,不再回來(無連鎖)
- 狼型:第 1 次 → **菁英化排末**(連戰場次 +1);精英狀態失敗 → GAME OVER
- 豹型 / 熊型 / 精英狼型 / 異變的巨狼(狼·終末形)精英狀態失敗:GAME OVER
- **異變的巨狼第一次失敗**:菁英化(等同狼型,失敗不直接 GO,**預期玩家首戰會敗**)

**參考**:[遊戲核心系統機制.md](遊戲核心系統機制.md) §5 + §17(教學重來)

### 3.5 戰役狀態機

**職責**:管理連戰 / 整備節點 / Hub 切換。

**序章前半結構**(第一期實作首要目標):

```
[Hub] 
  → [戰役選擇]
  → [序章前半開場劇情:獵人父子入林]
  → [連戰 1:兔子×3] (教學重來機制啟用)
  → [整備 1:完全補給]
  → [連戰 2:兔+狐×2] → [整備 2:+2 射箭/+2 投石]
  → [連戰 3:兔+狐+狼] → [整備 3:+2/+2]
  → [連戰 4:狼×3]   → [整備 4:+3/+3]
  → [連戰 5:兔×2+異變的巨狼(預期菁英化二戰)]
  → [序章前半末劇情:父親離開]
  → [序章後半]    (結構待設計)
  → [回 Hub + 紀念品觸發]
```

**核心類別**:
- `CampaignState`:當前戰役進度 + `tutorial_retry_enabled` flag + `retry_count`

```gdscript
@export var supply_type: String           # "full_restore" / "add_cards"

@export var narrative_id: String          # 對應的劇情段
```

### 3.6 失敗重啟機制(資源持續縮減模型)

**職責**:處理 GAME OVER 後的重啟邏輯。

**流程**:

```gdscript
func on_game_over(failed_at: BattlePosition):
    var choice = await prompt_player_choice()  # 重新挑戰 / 接受失敗
    match choice:
        "retry":
            _restore_to_prep_node(failed_at)
            # 卡組狀態保持失敗時的消耗,整備期不再次觸發
        "accept":
            _campaign_failed()
            # 回 Hub,可重新挑戰整場戰役(此時卡組重置)
```

**關鍵規則**:
- 卡組狀態保持(消耗的牌不回來)
- 整備期回補不重複觸發
- 連戰內狀態重置(狼精英化清除)
- 角色牌 / 爆發牌已用 = 已用

**參考**:[遊戲核心系統機制.md](遊戲核心系統機制.md) §7

### 3.7 戰鬥紀錄系統

### 3.13 教學重來機制(2026-05-14 新增)

**職責**:序章前半第一連戰失敗時,卡組與狀態回滾並重來。

> 對應 [遊戲核心系統機制.md](遊戲核心系統機制.md) §17。

```gdscript
# scripts/logic/tutorial_retry.gd
class_name TutorialRetry
extends RefCounted

static func on_chain_start(ctx: CampaignContext):
    if ctx.current_chain.tutorial_retry:
        ctx.tutorial_retry_enabled = true
        ctx.retry_count = 0
        ctx.deck_snapshot = DeckManager.snapshot()

static func on_rabbit_failed(ctx: CampaignContext):
    # 觸發連鎖逃跑 → 連戰結束 → 敘事覆蓋 → 回滾
    EventBus.emit_signal("tutorial_retry_triggered", ctx.retry_count)
    DeckManager.restore(ctx.deck_snapshot)
    ChainState.reset_chain(ctx.current_chain)
    ctx.retry_count += 1
    # 對話衰減分支由 DialogueController 依 retry_count 決定

static func on_chain_success(ctx: CampaignContext):
    ctx.tutorial_retry_enabled = false
    ctx.deck_snapshot = null
```

**對話衰減分支**(DialogueController 處理):
- `retry_count` 1-2:鼓勵性 + 弱點對應提示
- `retry_count` 3-4:中性「再試一次」
- `retry_count` 5+:略諷刺

**重要規則**:
- 教學重來在「整備 1 自動存檔」之前完成,**重來不存檔**
- 重來計數不顯示給玩家(內部用)
- 第一連戰成功後 flag 清除,連戰 2 起進入正式戰役規則

### 3.14 整備補給系統(2026-05-14 新增)

**職責**:依劇情階段固定補給卡組,無玩家選擇。

> 對應 [遊戲核心系統機制.md](遊戲核心系統機制.md) §9 + [卡牌資料庫.md](卡牌資料庫.md) 第四章。

```gdscript
# scripts/logic/supply_handler.gd
class_name SupplyHandler

static func apply(phase: SupplyPhase, deck: DeckManager):
    match phase.supply_type:
        "full_restore":
            # 用「戰前快照」恢復(僅整備 1 使用)
            deck.restore_to_prebattle_snapshot()
        "add_cards":
            for card_id in phase.cards_to_add:
                deck.add(card_id, phase.cards_to_add[card_id])
    EventBus.emit_signal("supply_applied", phase.id)
```

**序章前半補給序列**:
| Phase id | 觸發 | 內容 |
|---------|------|------|
| `rest_1` | 連戰 1 結束 | full_restore |
| `supply_2` | 連戰 2 結束 | `{ tool_arrow_pierce: 2, tool_stone_impact: 2 }` |
| `supply_3` | 連戰 3 結束 | 同上 |
| `supply_4` | 連戰 4 結束 | `{ tool_arrow_pierce: 3, tool_stone_impact: 3 }` |

**整備節點完整流程**:
1. 戰後校準對話(DialogueController)
2. 小劇情段落(NarrativeController)
3. 補給應用(SupplyHandler.apply)
4. 自動存檔(SaveSystem.save_at_prep_node)
5. 進入下一連戰

### 3.12 動畫 / 音效層

**職責**:OHK 動畫、結算 bar 填充、音效觸發。

**Godot 工具**:
- `AnimationPlayer` 用於 OHK 擊殺動畫、需求 bar 填充
- `Tween` 用於 UI 緩動
- `AudioStreamPlayer` 用於音效

**關鍵動畫**:
- OHK 擊殺:0.5-2 秒(可設置速度)
- 需求 bar 填充:0.3-1 秒
- 卡牌拖放:即時(無動畫,只有視覺反饋)
- Place → Lock 過渡:0.1 秒

**音效觸發點**:
- 卡牌 Place(輕)
- 卡牌 Lock(稍重)
- 結束本擊(明確)
- OHK 成功(儀式化)
- OHK 失敗(壓抑)
- 不同敵人類別的失敗音(兔逃 / 狼精英化 / 豹熊 GO)

## 四、資料格式規範

| 資料類型 | 格式 | 位置 |
|---------|------|------|
| 卡牌定義 | `.tres` Resource | `res://resources/cards/` |
| 敵人模板 | `.tres` Resource | `res://resources/enemies/` |
| 戰役配置 | `.tres` Resource + `.tscn` 場景 | `res://resources/campaigns/` |
| 整備補給規格 | `.tres` Resource | `res://resources/supply_phases/` |
| 對話文本 | `.csv` 或 `.tres` | `res://resources/dialogues/` |
| 紀念品定義 | `.tres` Resource | `res://resources/souvenirs/` |
| 存檔 | Godot `save_file()`(`.tres`) | `user://savegame.tres` |
| 戰鬥紀錄 | JSON | `user://battle_records.json` |
| 設置 | `ConfigFile` | `user://settings.cfg` |

### 為什麼用 .tres 而非 JSON

- Godot 編輯器內建可視化編輯 .tres
- Type-safe(編譯時檢查欄位)
- 引用其他 Resource 容易(card → special_effect)
- 但 JSON 仍保留為**外部工具產出格式**(`generate_card_excel.py` 之類)

### 與 Excel 工具的整合

現有 `tools/generate_card_excel.py` 從卡牌定義產出列印版 xlsx。未來可:
- 加一個 `import_from_json.py` 把 JSON 轉成 Godot `.tres`(或反向)
- 維持 Excel 作為「設計工作流」,Godot Resource 作為「實作格式」

## 九、舊專案參考速查表

| P6 模組 | 對應舊檔 | 參考價值 |

### Tier 2:結構參考(P6 機制不同,但結構可借鏡)

| P6 模組 | 對應舊檔 | 借鏡重點 |
|---------|---------|--------|

### Tier 3:不參考(P6 機制完全不同)

| 舊模組 | 為何不參考 |
|--------|----------|
| `DeckService` | P6 無抽牌 |

### 使用方式

實作 P6 各模組時,可以:

1. **先讀對應的 Tier 1 / 2 舊檔**,了解既有解法

3. **AI 翻譯 C# → GDScript**,並依 P6 設計差異調整(欄位、互動)
4. **若舊解法不適用 P6**,直接重新設計,不勉強

- **M0 結束前**:挑出 3-5 個重點檔案閱讀,標記為「實作時必看」
- **每個 milestone 開始時**:檢視該階段對應的舊檔,決定是否參考
- **遇到具體實作問題時**:用本速查表反查
