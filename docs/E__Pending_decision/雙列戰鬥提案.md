## 一、提案動機與目標

### 提案動機

P6 既有單列戰鬥(一場本擊對一個敵人)有兩個結構性限制:

1. **「對誰用」的策略張力薄弱**:在沒有提前揭露需求的情況下,玩家**不知道** → **無從選擇對誰用招牌卡** → 失去「卡牌分配」的策略樂趣
2. **BOSS 戰跟一般戰鬥本質相同**:目前 BOSS(熊型)跟一般敵人(狼型)的差異只在「**需求數值更高**」與「**失敗 = GAME OVER**」,**機制結構完全相同**

### 提案目標

| 目標 | 對應的玩家體驗 |
|------|-------------|
| **「對誰用」的策略張力** | 玩家面對招牌卡時,除了「**用不用**」還要思考「**對誰用**」 |
| **特殊敵人差異化** | BOSS 戰除了數值,還有**機制結構**的不同 |

### 限制

- **不是常態**——只在特殊戰鬥 / 特殊 BOSS 使用
- **第一期不實作**——維持單列戰鬥為基準
- **不過度複雜化**——保持 OHK 美學的簡潔精神

## 二、核心機制與設計前提

### 核心機制

```
一次本擊內:
  ┌─ A 列 ──────────────┐  ┌─ B 列 ──────────────┐
  │ 玩家分配卡到 A 列     │  │ 玩家分配卡到 B 列     │
  │ A 列獨立計算類型加總  │  │ B 列獨立計算類型加總  │
  │ A 列獨立比對需求表    │  │ B 列獨立比對需求表    │
  └─────────────────────┘  └─────────────────────┘
           ↓                           ↓
        同時結算(一個本擊的結束本擊瞬間)
           ↓                           ↓
        A 列達標/失敗            B 列達標/失敗
           ↓                           ↓
        各自依「目標類型」判定後果
```

### 設計前提(用戶確認)

| # | 前提 | 必須符合 |
|---|------|--------|
| 1 | **同時結算** | 不是先 A 後 B,是一個本擊瞬間判定兩列 |
| 2 | **各敵人按類型獨立判定失敗** | 該逃的逃(兔)、該精英的精英(狼)、該 GO 的 GO(豹/熊) |
| 3 | **打出的卡不能回收** | 跟既有 Lock 機制一致 |
| 4 | **戰鬥條件跟原本一樣** | 仍是類型計數結算,任一條件達標即 OHK |
| 5 | **一個敵人仍只有一次機會** | 等於把兩回合的選擇壓成一回合 |
| 6 | **不是常態** | 只在特殊戰鬥(BOSS、多部位敵人、團體戰) |

### 跟單列的本質差異

| 面向 | 單列戰鬥 | 雙列戰鬥 |
|------|--------|--------|
| 一個本擊面對的目標 | 1 | 2 |
| 玩家的決策數 | 「用什麼卡」 | 「用什麼卡」**+**「對誰用」 |
| 結算次數 | 1 | 2(並行) |
| 失敗代價 | 單一(該敵人類型) | 多重(各敵人類型獨立) |
| 視覺呈現 | 單敵人立繪 | 雙敵人立繪 |

## 三、跟 P6 既有設計的相容性分析

### ✓ 跟「OHK 美學 / 瞬間即儀式」

[遊戲設計概念.md](../B__Design_specifications/遊戲設計概念.md) §3 原則三「**瞬間即儀式**」:
> 那一擊本身就是獎勵。

雙列下:
- 一個本擊 = 玩家一次大決策(分配兩列)
- **結算瞬間是「同時兩擊」** — **是另一種儀式感**(像武士同時砍倒兩個敵人)
- 不是被稀釋,是不同的演出形態

**前提**:結算演出必須**同步**,不能「先 A 後 B」拆成兩個瞬間。

### ✓ 跟「單一檢查原則」

[遊戲核心系統機制.md](../C__Implementation_benchmarks/遊戲核心系統機制.md) §2:
> 勝敗判定收斂到一個結算瞬間。

雙列下:
- A 列達標 + B 列達標 = **同一個結算瞬間**
- 不是「先打贏 A,然後打 B」的 AND 結構
- 是「**同一本擊內的兩個並行判定**」

→ **不違反**。

### ✓ 跟「玩家是判斷者,系統是計算者」

雙列下:
- 玩家**判斷**「這張致命凝視該用在 A 還是 B」
- 系統**計算**兩列各自的類型加總
- 系統**不**推薦「哪列該用哪張」

→ **不違反**,只要 UI 各列獨立呈現加總(玩家不用心算)。

### ✓ 跟「失敗代價系統(四類差異化)」

[遊戲核心系統機制.md](../C__Implementation_benchmarks/遊戲核心系統機制.md) §5 的四類失敗代價在雙列下**完全相容**:

| 情境 | A 列敵人 | B 列敵人 | A 結果 | B 結果 | 整體後果 |
|------|--------|--------|------|------|--------|
| 1 | 狼 | 兔子 | 達標 | 達標 | 兩個都擊殺 |
| 2 | 狼 | 兔子 | 失敗 | 達標 | 狼變精英排末 + 兔擊殺 |
| 3 | 狼 | 狼 | 失敗 | 失敗 | 兩隻都變精英(連戰場次 +2) |
| 4 | 豹 | 狼 | 失敗 | 達標 | **GAME OVER**(豹失敗) |
| 5 | 兔子 | 兔子 | 失敗 | 失敗 | 兩兔逃走,卡牌消耗 |

→ **完全相容**,只是「同一本擊」處理多個結果。

### ✓ 跟「Place / Lock 機制」

雙列下:
- **Place 階段**:玩家可在兩列間移動卡
- **Lock 階段**:Lock 後卡固定在該列
- **結束本擊**:所有 required 卡都 lock 後,兩列同時結算

→ **不衝突**。實際上「**移卡到 A 還是 B**」就是 P6 既有 Place 階段判斷張力的延伸。

### ⚠ 跟「資源持續縮減失敗模型」需注意

雙列下:
- 一個本擊**消耗可能加倍**(兩列卡都用)
- 失敗時資源縮減更快

**緩解**:
- 雙列戰鬥的**敵人配對是設計者預設**(不是玩家隨機遭遇)
- 應在**卡組充足的時刻**觸發(連戰前段 / 特定 BOSS 戰)
- 雙列戰鬥**前**可有 NPC 對話 / 環境暗示,讓玩家心理準備

### ⚠ 跟「冒險手記」需要頁面重設計

目前戰鬥節點頁:左頁 = 使用的卡,右頁 = 敵人。

雙列戰鬥要呈現:
- A 列使用的卡 + A 列敵人
- B 列使用的卡 + B 列敵人

**可能解法**:
- **四象限頁面**(上半 A 列、下半 B 列)
- 或**雙頁變雙倍**(連續兩頁,標明「同一場雙列戰鬥」)

→ 可解決,設計工作量適中。

### ⚠ 跟「視覺暗示傳達(層 2 / 層 3)」認知負擔

雙列下兩個敵人同框,玩家**同時讀兩個敵人視覺**。

**緩解**:
- 雙列戰鬥的敵人**至少有一個是已熟悉類別**(老朋友)
- 或**故意設計成同類型**(兩隻巡守)減少認知負擔
- 或**對比類型**(一狼一兔)讓差異明顯

## 四、應用情境

### 情境 A:**雙敵人戰鬥**(兩個獨立敵人同框)

| 配置 | 設計感 |
|------|------|
| **BOSS + 護衛**(熊 + 狼) | 玩家要決定「先處理護衛還是直接打 boss?」 |
| **同類雙敵**(兩隻巡守) | 玩家分配資源平均處理,或一邊重一邊輕 |
| **誘餌 + 真兇**(兔 + 豹) | 玩家可放棄誘餌、集中對付真兇 |
| **對抗型雙敵**(兔 + 狼) | 兔可放、狼必過,玩家策略性放棄一個 |

### 情境 B:**單敵人多部位**(boss 有多個攻擊目標)

例如:**巨大機器人**,有「致命左手」與「不會致命的右手」。

| 列 | 部位 | 戰鬥意義 |
|---|------|--------|
| **A 列** | 致命要害(必擊) | 達標 = OHK 本敵 / 失敗 = 整隻擊不倒 |
| **B 列** | 非致命部位(可選) | 達標 = 完美擊殺(紀念品) / 失敗 = 部位殘存(可能延續到下場戰役) |

#### 「致命/可選列」的失敗結果設計

| 情境 | 結果 |
|------|------|
| A 達標 + B 達標 | **完美擊殺**(解鎖紀念品)|
| A 達標 + B 失敗 | **主體擊殺 + B 部位殘存** → 可能對應追擊機制延伸 |
| A 失敗 + B 達標 | **整體未擊倒** → 依敵人類別判定(熊型 = GO) |
| A 失敗 + B 失敗 | 同上(熊型 = GO) |

#### B 列失敗的三種設計選項

| 選項 | 機制 | 特性 |
|------|------|------|
| **純獎勵** | B 達標 = 紀念品;B 失敗 = 無事 | 鼓勵性,完美擊殺是 bonus 不是負擔 |
| **戰術回饋** | B 達標 = 連戰下個敵人狀態改變(被襲變先攻等)| 有意義的選擇,需計算長期效益 |
| **懲罰性** | B 失敗 = 殘存部位下場戰役回來追擊(精英化) | 跟失敗代價系統相容(對應狼型「失敗變精英」精神) |

**推薦混合**:**純獎勵 + 懲罰性**——B 達標解鎖紀念品,B 失敗的「殘存部位」會在後續以精英姿態出現。

### 情境 C:**團體敵人** / **小隊遭遇**

- 一場本擊面對「一個小隊」
- 兩列代表不同優先級的敵人

## 五、UI 設計挑戰與選項

### 選項 1:**本擊區水平分上下兩列**

```
┌─ 上方戰場 ─────────────────────┐
│ 敵人 A 立繪 + 敵人 A 資訊       │
└─────────────────────────────────┘
─ 上列本擊區 ─────────────────────
[卡] [卡] [卡]    本擊狀態 A
─ 下列本擊區 ─────────────────────
[卡] [卡]         本擊狀態 B
┌─ 下方戰場 ─────────────────────┐
│ 敵人 B 立繪 + 敵人 B 資訊       │
└─────────────────────────────────┘
```

| 優 | 缺 |
|---|---|
| 兩列同時可見 | 敵人立繪縮小 |

### 選項 2:**本擊區水平分左右兩列**(推薦)

```
┌─ 兩個敵人同框(縮小)──────────┐
│ 敵人 A         敵人 B          │
└────────────────────────────────┘
─ 本擊區左右分區 ──────────────────
[A 列 ←─卡卡卡──→][B 列 ←─卡卡卡──→]
本擊狀態 A         本擊狀態 B
```

| 優 | 缺 |
|---|---|
| 類似目前單列水平布局 | 卡牌縮略空間減半 |
| 「對誰用」視覺最直接(拖到 A 或 B) | |

### 選項 3:**前後列切換(tab 式)**

```
[A 列][B 列] ← tab 切換
═══════════════════════
顯示當前 tab 的本擊區(全寬)
═══════════════════════
庫存區
```

| 優 | 缺 |
|---|---|
| 每列 UI 寬敞如單列 | 看不到另一列當下狀態,認知負擔重 |

### 推薦:**選項 2(左右分區)**

理由:
- 同時可見性最強
- 「對誰用」視覺最直接(拖卡到對應列)
- 玩家看得到當下分配狀態
- 失去的卡牌空間可透過縮略 + 點擊展開補償

### 額外 UI 元素

- **本擊狀態雙顯**:A 列加總 + B 列加總獨立顯示
- **結束本擊按鈕**:只有一個(觸發兩列同時結算)
- **required 卡 lock 檢查**:必須兩列所有 required 卡都 lock 才能結束本擊

## 六、結算呈現順序設計

雙列戰鬥的關鍵問題:當 **A 列觸發 GO 而 B 列達標**時,如何呈現?

### 選項 A:**同步呈現,GO 最後**(推薦)

```
雙列同時演出:
  A 列:豹型擊殺動畫嘗試 → 失敗(豹未達標)
  B 列:狼型擊殺動畫 → 成功
  ↓
  (兩列演出完成)
  ↓
  「但豹型失敗 = GAME OVER」彈出
```

**好處**:
- 玩家先看到「擊殺成果」(若有)
- 再面對 GO 的重擊
- 「達標但仍 GO」的悲劇感更強(像「贏了戰役,輸了戰爭」)
- 對應「失敗有真實代價」的設計精神

### 選項 B:**GO 立即中斷**

```
雙列同時演出開始:
  A 列:豹型擊殺動畫 → 失敗
  ↓ 立即
  GAME OVER 彈出
  ↓
  B 列(若達標)在 GO 畫面後簡短呈現
```

**好處**:GO 的儀式感不被稀釋。

### 推薦:**選項 A**

雙列戰鬥的核心是「**多重結果同時呈現**」,GO 應該是**最終一擊的儀式**,不是中斷。

## 七、失敗代價系統整合

### 跟既有四類失敗代價的整合

所有失敗代價邏輯**保持不變**,只是在雙列中**並行套用**:

| 列敵人類型 | 列失敗時 |
|---------|--------|
| **兔子型** | 該兔逃走,卡消耗 |
| **狼型** | 該狼變精英排末(整體連戰場次 +1)|
| **狼型(精英狀態)** | **GAME OVER**(整場) |
| **豹型** | **GAME OVER**(整場) |
| **熊型** | **GAME OVER**(整場) |

→ **規則完全不變**,只是「**整場 GAME OVER**」的觸發條件變成「**任一列觸發 GO**」。

### 跟「資源持續縮減失敗模型」整合

雙列戰鬥失敗後:
- 玩家選「重新挑戰」 → **整個雙列戰鬥重打**
- 卡組保持失敗時的狀態(消耗已扣)
- 連戰內狀態重置(失敗的狼若是 B 列的,精英化清除)

→ **跟既有重啟邏輯相容**。

### 多部位敵人(B 列「部位殘存」)的延伸

對應情境 B(boss 多部位)的「**B 列失敗 → 部位殘存追擊**」:

```
玩家擊敗巨大機器人(A 達標),但右手未致殘(B 失敗)
  ↓
進入下個整備節點
  ↓
下場戰役 / 本連戰末段
  ↓
「殘存的右手機器」以小型精英姿態出現
  ↓
玩家需用殘餘資源擊敗
```

→ 對應 P6 既有的**狼型精英排末**機制的概念延伸。

## 八、多部位 boss 的延伸應用

### 設計原則

| 原則 | 內容 |
|------|------|
| **2 列極限** | 機制層保持 2 列(避免認知負擔失控);多部位**在敘事 / 視覺層**呈現 |
| **致命/可選明確** | A 列 = 致命必擊,B 列 = 可選次要 |
| **B 列有明確獎勵與懲罰** | 既有 carrot 也有 stick,讓「對誰用」的決定有重量 |

### 範例:**巨大機器人 boss(熊型)**

```
敵人:巨大機器人(熊型,劇情 BOSS)

A 列「致命核心」:
  弱點:衝擊 / 穿刺
  需求:衝擊 6 / 穿刺 9 / 混合 24
  失敗:GAME OVER(熊型)

B 列「武裝右手」:
  弱點:穿刺 / 燃燒
  需求:穿刺 4 / 燃燒 4 / 混合 12
  失敗:「機械殘骸右手」精英化,在追擊階段出現
       (對應狼型精英化排末邏輯)

完美通關條件:A + B 都達標
紀念品:「機械殘骸的徽章」(完美通關才出現)
```

### 視覺呈現(多部位但 2 列)

巨大機器人立繪上有**多個視覺焦點**(左手 / 右手 / 雙腳 / 核心),但機制上聚焦兩列:
- **A 列覆蓋**:核心 + 左手 + 雙腳(代表整體生命力)
- **B 列覆蓋**:右手(代表附加武裝)

→ 視覺豐富,機制精簡。

## 九、跟其他被拒絕提案的對比

### 雙列戰鬥 vs 分階段需求變化(被拒絕)

| 比較 | 分階段需求(拒) | 雙列戰鬥(本提案) |
|------|---------------|----------------|
| 結構 | 連續本擊(打到 X% 進階段 2)| 同時結算(一次本擊雙列) |
| 是否變相 HP | **是**(等同連續扣血條) | **否**(單一決定瞬間) |
| 違反 OHK | **是**(多次擊) | **否**(同時兩擊仍是一個瞬間) |
| 「對誰用」張力 | **無** | **有** |
| 視覺差異化 | 弱(只是更多血) | 強(部位 / 雙敵呈現) |
| 跟失敗代價系統 | 衝突(難判定階段失敗)| 完全相容(各列獨立)|

→ **雙列戰鬥是更好的設計**。

### 雙列戰鬥 vs 部位修飾(單一檢查 + 視覺包裝)

| 比較 | 部位修飾(視覺包裝) | 雙列戰鬥 |
|------|---------------|-------|
| 機制 | 單一檢查,部位是視覺解釋 | 兩列獨立計算 |
| 玩家「對誰用」 | **無**(實際上跟普通敵人一樣) | **有** |
| 設計複雜度 | 低 | 中 |
| 適用情境 | 一般敵人的視覺豐富化 | 特殊 boss / 雙敵情境 |

→ **兩者可共存**。普通敵人用「部位修飾」即可;特殊 boss / 多目標才用雙列。

---

## 十、何時不該採用

### 不適合雙列的情境

| 情境 | 為什麼不適合 |
|------|----------|
| **教學戰役(序章)** | 玩家還在學單列機制,雙列認知負擔過重 |
| **一般狼型 / 兔型敵人** | 機制差異化無意義,反而違反「視覺即敘事」原則 |
| **資源不足的連戰段** | 雙列消耗加倍,容易讓玩家絕望 |
| **追擊階段的精英敵人** | 應該保持單對單的儀式感 |
| **常態化** | 雙列若變成每場都是,就失去「特殊性」 |

### 適合雙列的情境

| 情境 | 為什麼適合 |
|------|----------|
| **後期戰役的 BOSS 戰** | 差異化 boss 體驗 |
| **多部位巨型敵人** | 致命/可選列的策略張力 |
| **故事高潮的雙重對峙** | 敘事與機制同步加強(例:對抗師徒、雙重身份反派) |
| **特殊事件戰鬥** | 連戰中段的「意外」雙敵遭遇 |
