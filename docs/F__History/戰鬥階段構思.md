## 一、核心結構:單次 commit 戰鬥

### 基本流程
每場戰鬥就是**一個 commit moment**:玩家從手上挑選若干張卡組成「本擊」,一次提交,所有效果同時結算。

```
敵人登場 → 玩家組成本擊 → 一次性結算 → 成 / 敗
```

### 為什麼不是兩段式
強制性的「鋪陳階段 + 一擊」會稀釋 OHK 美學。OHK 的精髓是「單一決定性瞬間」,一旦把戰鬥分成多個小決策,那個瞬間就被攤散了。

### 玩家體驗的時間結構
- **Commit 之前**:判斷
- **Commit 之後**:揭曉
- **中間**:沒有任何拖延

### 「本擊」的定義
本擊 = 玩家在單場戰鬥中,從手上挑出並一次性提交的卡牌組合。

本擊是設計上的最小單位,不是個別卡牌:
- 揭露效果依「本擊」內的卡綜合判定
- 結算結果依「本擊」整體類型次數決定
- 玩家的決策對象是「本擊」而非「個別卡」

## 二、結算分層(Place / Lock / 結束本擊)

本擊雖然是「單一 commit 階段」,內部結算分為三個子階段。Place/Lock 機制的完整定義詳見 [卡牌構思.md 第七章 本擊概念與 Place/Lock 機制](卡牌構思.md)。

### 卡牌的 lock_class 三分類(2026-05-11)

每張卡的 `lock_class` 決定它如何與本機制互動:

- **`"none"`** —— 純戰鬥卡(資源池、純戰鬥爆發/角色)。Lock 與否不影響效果。
- **`"optional"`** —— 情報卡(偵察/估量/警示)。不 lock = 後續廣度揭露;Lock = 當前深度揭露(犧牲廣度)。
- **`"required"`** —— 試探卡、flexible 爆發/角色、洞察一擊、全知之眼。**必須先 lock 才能生效**;「結束本擊」前所有 required 卡必須先 lock。

### 三個子階段

1. **Place 階段**:玩家拖卡到本擊區,可隨時編輯。系統不結算任何效果,不消耗資源。flexible 卡顯示「未分配」狀態。

2. **動態 Lock 階段**:玩家對單張卡點「鎖定」:
   - **`"none"` 卡**:可 lock 也可不 lock,效果不變
   - **`"optional"` 卡**:lock 觸發 locked 模式;不 lock 則於「結束本擊」觸發 unlocked 模式
   - **`"required"` 卡**:**必須在此階段 lock**(否則「結束本擊」按鈕禁用)。lock 時 special_effect 立即觸發,flexible 必須完成分配
   
   Lock 不可撤回。Lock 過程中玩家可繼續新增 placed 卡或 lock 其他卡。

3. **結束本擊**:玩家點「結束本擊」按鈕(若有 required 卡未 lock 則禁用),系統鎖定所有未 lock 的 a/b 類 placed 卡,然後在戰鬥層整體計算各類型次數總和,逐項比對敵人需求表;**任一條件達標即 OHK**。
   - 期間若 b 類卡未 lock,觸發其 unlocked 模式 special_effect

### 結算範例

敵人需求表:衝擊 2 / 穿刺 5 / 燃燒 6 / 混合任意 7

玩家本擊流程:

```
1. [Place] 玩家置入:洞察一擊、重斬、直擊
   本擊狀態:衝擊 0(尚未分配 + 未結算)

2. [Lock 洞察一擊]
   → 觸發 special_effect:揭露當前敵人弱點 = 衝擊
   → 洞察一擊資源消耗(角色卡用掉)
   → 玩家現在知道弱點

3. [玩家分配洞察一擊的 flexible]
   洞察一擊 +2 → 分配 +2 衝擊

4. [結束本擊]
   → 鎖定剩餘:重斬、直擊
   → 戰鬥層整體計算
   → 重斬 +2 衝擊、直擊 +1 衝擊、洞察 +2 衝擊
   → 衝擊計數 = 5(需求 2)→ ✓ 達標
   → 結果:OHK
```

### 設計含義

Place/Lock 機制讓「揭露」與「決策」之間有自然空隙——玩家做出**多個子動作**,但這些子動作仍構成「**一個本擊**」。

**這跟「分次戰鬥」是兩回事**——玩家是在同一 commit 階段內讓資訊流動,不是發起多次 commit。對應 OHK 美學的「物理動作鏈」精神:一旦動作開始(lock)就不可收回,但其他動作可以基於資訊調整。

### Lock 順序的策略意義

Lock 順序本身是判斷的一部分:
- 先 lock 揭露卡 → 獲取資訊優勢 → 後續決策更精準
- 先 lock 戰鬥卡 → 喪失資訊優勢,但表達「我已決定」的清晰承諾

設計上不限制順序,讓玩家自己經營「資訊→決策」的節奏。

## 三、戰鬥狀態:先攻 / 標準 / 被襲

每場戰鬥開始時的處境本身是個戰術變數。

### 先攻戰鬥
- **效果**:玩家獲得免費的前置揭露(一張情報牌等價的效果,不消耗手牌)
- **適用**:連戰第一個敵人(補正首回合資訊劣勢)、容易被偵察的敵人
- **設計感**:玩家有準備、佔上風

### 標準戰鬥
- **效果**:直接 commit
- **適用**:大部分敵人
- **設計感**:基本對峙

### 被襲戰鬥
- **效果**:部分情報效果無效,或必須投入更多卡牌,或限制可用卡片類型
- **適用**:埋伏者、刺客類敵人
- **設計感**:措手不及

### 為什麼這個分類有用
讓敵人類型不只有「需求表多少」這個維度,還有「如何進入戰鬥」這個維度,敵人個性立刻豐富。

### 待決定
- 戰鬥狀態是依敵人類型固定?
- 還是依玩家某些選擇影響?
- 或是隨機(input randomness 意義上)?

## 四、「猜弱點」機制

這是讓情報變成「可選工具而非必要工具」的關鍵設計。

### 核心邏輯
弱點是**客觀存在的事實**,揭露與否只影響玩家的「知曉」,不影響弱點本身。

### 玩家的選擇
- **A. 用情報換確定性**:花情報牌揭露弱點,下一場確定打中
- **B. 用元知識+直覺賭**:根據敵人外觀、行為線索,直接猜測攻擊類型

### 結算邏輯
- 出卡正確匹配弱點 → OHK 成立(無關玩家是否知道)
- 出卡未匹配弱點 → 失敗

### 情報的時間性
情報牌揭露的效果**延伸到下一場(甚至跨數場)的目標**,而不是為當前敵人鋪陳。這意味著:
- 情報是「投資」,不是「鋪陳」
- 玩家在 commit 時的決策變成「我這擊要不要塞情報牌?我這場可能因此打不贏,但下一場會輕鬆很多」
- 這是用當下的成功率買未來的確定性

### 為什麼這個機制好
- 情報變成可選工具而非必要工具
- 「勇敢的玩家」可以一直賭,「謹慎的玩家」可以一直查
- 兩種玩法都成立,定義玩家性格
- 完美對應 OHK 美學:工具不是必需品

### 設計責任
這個機制成立的前提是**敵人外觀必須提供足夠視覺線索**。盔甲類型暗示防禦特性、姿態暗示弱點位置、配色暗示屬性傾向。美術設計與遊戲機制要緊密配合。

## 五、失敗代價(2026-05-11 新機制)

OHK 美學要求失敗必須痛但不能無趣。完整機制詳見 [**失敗代價與精英機制.md**](失敗代價與精英機制.md)。

### 核心邏輯

> **並非所有敵人都會殺你——但會記恨的敵人會以精英形態回歸,而預設精英與 Boss 失敗即終戰。**

失敗代價**依敵人類別**(兔子 / 狼 / 豹 / 熊)差異化:

### 依敵人類別的失敗代價摘要

| 類別 | 範例 | 失敗代價 |
|------|------|---------|
| **A. 兔子(無威脅普通)** | 哨兵 | 卡牌消耗,敵人逃走不再回來 |
| **B. 狼(威脅普通)** | 巡守、重甲衛兵 | 卡牌消耗 + 變精英排末 / 累積 2 次 → 進未解決池 + 追擊選擇 + 替換下連戰 |
| **C. 豹(預設精英)** | 伏擊者 | **失敗即 GAME OVER** |
| **D. 熊(Boss)** | 領主 | **失敗即 GAME OVER** |

### 設計原則

1. **敘事邏輯一致**:兔子不會記恨、狼會記仇、豹本來就要殺你、熊是必殺局
2. **失敗代價漸進可感知**:從輕(兔子)→ 中(狼第一次)→ 重(狼累積)→ 終(豹/熊)
3. **隱性戰役失敗條件**:不用 Lives 系統,但精英失敗 = GO 提供有機的「戰役結束」機制
4. **追擊機制提供玩家自主性**:連戰結束後可選擇追擊或繼續

### 完整流程、視覺要求、UI 設計

詳見 [失敗代價與精英機制.md](失敗代價與精英機制.md)。

## 六、跳過機制

### 實作方式
做成有限次數的卡片(例如「迴避」、「轉進」),不是隨時可用的選項。

### 為什麼用卡片形式
- 跳過有機會成本(這張卡用了就沒)
- 避免養成「遇到不確定就跳」的逃避習慣
- 跳過本身可以是有趣的設計題

### 跳過卡的設計變化
- 一張只能跳當下
- 一張可以跳隊列中任何一個
- 一張跳過會讓敵人變強
- 一張跳過後該敵人在最終戰會出現

### 重要設計原則
**最終連戰中無法跳過,前期連戰可以跳過但有累積代價**——把跳過變成「拖到後面再算總帳」的策略選擇,而不是「逃避困難」的取巧。

## 七、敵人反應與互動

### 設計原則
每個敵人類型應有一個**簽名反應**,讓敵人辨識變得有意義。

### 反應類型
- **被動反應**:暴露更多狀態(被弱化會露出原本看不見的弱點)
- **主動反應**:強化自身(被偵察過久會進入警戒,需求次數上升)
- **環境反應**:影響後續戰鬥(這敵人的死法會改變下一個敵人)

### 為什麼重要
沒有互動感的敵人就只是 HP 條的具象化。簽名反應讓玩家對敵人類型形成記憶,認得出來的玩家會選擇對應策略,不認得的玩家會被反噬。

## 八、戰鬥之間的銜接

### 為什麼需要
單次 commit 結構下,每場戰鬥獨立。但連戰必須在戰鬥之間有銜接,否則只是「五場獨立戰鬥」而非「真正的戰役」。

### 可選銜接機制
- **資源延續**:前一場剩下的某些效果可延續到下一場開頭
- **連殺累積**:連續 OHK 累積某種狀態,影響下一場
- **敵人連動**:某些敵人是前一場的「同伴」,前一場的擊殺方式影響它們的初始狀態
- **資訊共享**:前一場揭露的某些情報屬於該類型,下一場遇到同類型可套用

### 設計建議
不一定全部要做,**但至少選一個**,否則連戰只是表面的,骨子裡還是獨立戰鬥的拼接。

## 九、戰後校準系統

### 系統目的(三個層次)

**第一層:校準直覺**
讓玩家知道這次決策大幅超出、剛好夠、還是差在哪裡——把「直覺判斷」與「實際結果」對照,逐步校準對敵人強度與卡牌效益的感知。

**第二層:指向原因**
不直接說「你哪裡錯了」,而是給方向性的提示讓玩家自己連結。校準的不是答案,是讀題能力。

**第三層:補充視覺辨識的教學負擔**
特別是在玩家還沒建立起讀層 2 線索的能力時,校準對話可以當補充管道,把「敵人外觀透露了什麼」這件事用語言指出來。等玩家熟練後,這層自然減弱。

### 呈現方式:對話而非數值

校準系統採用**雙聲道對話 + 敵方反應**的形式,不直接顯示數字。

精確資訊(各類型次數貢獻明細、精確需求表、達標路徑、跨場統計)放在「**戰鬥紀錄**」中(玩家可主動查閱),不出現在即時校準回饋中。

這個選擇是「呈現層原則」(設計概念十二章)的完整實踐——數值版本的校準畫面其實是對該原則的退讓,對話版本完整堅守「系統是計算者,玩家是判斷者」。

### 雙聲道架構

戰鬥內限定兩個聲音:玩家角色 + 單一 NPC。排除多角色合唱——單次 commit 結算下,所有角色都在旁觀同一件事,多人同時評論會聲音分工不清。

**玩家聲音 — 決策視角**
代表玩家在反省自己的估算。
- 範例:「我低估了那層護甲」「比想像的脆」

**NPC 聲音 — 觀察視角**
代表旁觀分析者,提供玩家可能沒察覺的角度。
- 範例:「那個持矛的動作其實有透露,你看出來了」「護盾沒偵測到,下次要注意」

兩個聲音天然形成「你怎麼看 / 我怎麼看」的張力,不需要調度就能產生節奏。

### 第三條反饋通道:敵方反應

敵方反應在結算動畫中即時呈現,獨立於對話之外。

**反應類型**
| 結果 | 反應 | 設計用意 |
|------|------|---------|
| 弱點被讀對的 OHK | 短暫驚呼或憤怒——「怎麼知道那裡」 | 讓「精準擊殺」與「蠻力擊殺」在感受上不同 |
| Underkill,差一點 | 緊繃、沉默 | 玩家感覺「就差一點」 |
| Underkill,差很多 | 反而輕鬆,甚至嘲諷 | 敵人越輕鬆,玩家越知道自己低估了多少 |
| 大幅 Overkill | 來不及反應,直接崩解 | 「沒有反應」本身就是壓倒性的回饋 |

**與「對稱即誠實」的關係**
驚呼觸發條件是「達標路徑為弱點類型(該類型最低需求)」,**不是「玩家事先知道」**。蠻力擊殺(走混合任意路徑達標)即使 OHK 成功也不該觸發驚呼——這保留了「老手識讀 vs 蠻力解決」的感受差異。

但完整的觸發條件需要進一步研究——尤其要區分:玩家是否曾主動揭露弱點?還是 commit 結果偶然對應?這影響敵方反應作為「情報回饋」而非「免費資訊獲得點」的設計平衡(列入待解決問題)。

### 6 種對話狀態

對應戰鬥結果的區間。語氣本身就是信號,玩家會學會讀「這個語氣 = 這個結果類型」,進入無需文字解說的直覺化閱讀層次。

| 結果區間 | 對話基調 | 指向的學習 |
|---------|---------|-----------|
| 大幅 Overkill | 輕鬆、甚至有點浪費的語氣 | 「這隻可以省資源」 |
| 精準擊殺 | 滿足、穩定的語氣 | 「這個判斷記住」 |
| 勉強過 | 緊繃、鬆一口氣 | 「這次很險,剛好夠」 |
| 差一步(類型錯) | 有點困惑、重新分析的語氣 | 「弱點沒匹配到」 |
| 差一步(量不足) | 直接的不夠感 | 「資源再多一點就過了」 |
| 大幅不足 | 明確失策的語氣 | 「方向根本錯了」 |

### 發言順序

採用**統一規則**而非依結果條件切換:

- **敵方反應**:永遠是結算動畫的一部分(驚呼、沉默、嘲諷都是動畫節奏的延伸)
- **玩家 + NPC 對話**:永遠在動畫結束後(response 性質)

這樣敵方反應不是「對話的第一個發言者」,而是「事件本身的一部分」,呈現更穩定。

註:敵方反應的具體風格可能列入敵人模板的基本設計,讓不同敵人有簽名反應(列入待解決問題)。

### 對話可傳達 vs 不可傳達

對話格式有天然限制——這跟本作的設計方向完全配合(不希望玩家接受數值)。

**對話可傳達**
- 結果的「份量感」——「輕鬆過了」vs「勉強夠用」vs「差太遠」,靠語氣分辨
- 原因的「方向」——「鎧甲太完整,衝擊不夠用」vs「力道足夠但方向錯了」
- 情境的「記憶錨」——讓這場戰鬥在玩家記憶裡有特色,不是模糊的「我又打了一個衛兵」

**對話不適合傳達**
- 精確數值
- 具體的卡牌效率比較
- 跨場統計

→ 這些放在可主動查閱的「戰鬥紀錄」中,不出現在即時回饋。

### 對話量策略

採用**「核心儀式化短語 + 模板識別性長對話」混合策略**,目標總量約 40–50 句(可手寫,不需動態生成)。

- **核心 6 句**:每種對話狀態 1 句經典固定台詞,跨敵人共用。重複本身成為 OHK 美學的一部分(類似 Dark Souls 的「YOU DIED」)
- **每模板 1–2 句**:每個敵人模板有專屬識別性長對話,只在「精準擊殺」與「大幅不足」這兩個感受最強的狀態觸發。讓老玩家形成「這種敵人 + 這種結果 = 這句話」的記憶錨

### 自適應教學

NPC 教學深度依**連戰位置**調整(粗粒度,以連戰為單位):
- 早期(首輪連戰):直接點出視覺線索
- 中期:給方向性提示
- 最終連戰:幾乎沉默,讓壓迫感自己說話

**框架詮釋**:不是難度調整,是「角色觀察玩家成長」——因為說話的是角色而不是提示框,說多說少可以被詮釋為「角色與玩家的關係變化」。

詳細的教學節奏屬於**整體戰役配置**範疇,本章不展開。

### 與其他系統的關係

- **與情報牌**:校準對話揭露的資訊不會延續到下一場(已死敵人的資訊無延續價值)。情報牌仍然是揭露未來敵人的唯一管道
- **與設計合約**:每個敵人模板的「設計合約-新手失敗時能理解什麼」依賴此系統存在。沒有校準對話,大部分模板的合約無法兌現
- **與紀錄流**(設計概念六章):精確數值與分析資料歸檔到戰鬥紀錄,玩家可主動查閱

### 待解決的問題

1. **NPC 性格與關係**(最關鍵)——決定整個對話系統聽起來像什麼樣的人在說話。候選方向:沉默的同行者 / 冷峻的記錄者 / 務實的副手 / 同伴靈獸熟靈
2. **敵方反應的觸發條件**——「攻擊類型對應弱點」要進一步區分:是否需要玩家曾主動揭露?還是看 commit 結果即可?影響「情報回饋 vs 免費資訊」的平衡
3. **敵方反應的個體化**——每個敵人模板是否需要簽名反應風格?若是,需擴充敵人模板格式新增「敵方反應」欄位
4. **對話時機**——結算動畫期間(同步呈現)還是動畫結束後(序貫呈現)?
5. **老玩家跳過機制**——預設顯示,玩家手動關閉(代價是學習效率降低)?還是依連戰位置自動減少對話?
6. **整備階段的多角色聲音**——戰鬥內限定雙聲道,整備期是否引入更多聲音?屬於更大的敘事結構問題

## 十、視覺語言要求

由於玩家可以「直接猜攻擊類型」,敵人的外觀必須提供足夠提示。

### 必須要做
- 同類敵人有共同視覺特徵
- 弱點類型可從外觀推論
- 戰鬥狀態(先攻 / 被襲)有視覺暗示

### 美術與機制的綁定
這不只是美學問題——遊戲機制本身依賴視覺溝通。**美術設計與機制設計必須同步進行**,不能分開做。

### 擊殺瞬間的儀式化
OHK 沒有經驗值或裝備獎勵,**「這一擊本身」就是獎勵**。如果擊殺感平淡,整個遊戲的核心體驗就垮了一半。

擊殺瞬間應該包含:
- 視覺強化(慢動作、特殊效果)
- 音效設計(明確的擊殺音)
- 節奏停頓(讓玩家「看見」勝利的瞬間)
- 不能讓玩家投牌後立刻看到結果——要讓他停頓一秒,看著各類型需求 bar 被填充逼近達標線,然後才崩解

---

## 十一、待解決的設計問題

1. **手牌規模**:玩家手上能持有多少卡?合適範圍可能 6–10 張,需實機測試
2. **本擊規模**:單次 commit 的卡片數量上限?3 張?5 張?無上限但有間接限制?
3. **先攻 / 被襲的判定邏輯**:依敵人類型固定?依玩家選擇影響?
4. **跳過卡的具體機制**:種類、限制、銜接後續戰鬥的方式
5. **儀式化的具體呈現**:擊殺瞬間的視覺、音效、節奏設計
6. **連戰中的銜接機制**:採用哪一種或哪幾種

## 設計鏈下一步

戰鬥結構的骨架已經穩定。下一步比較關鍵的是「**敵人的個性設計**」——既然戰鬥已經被壓縮成單次 commit,敵人的差異性就必須完全靠以下維度體現:

- 需求表(各類型次數 + 混合任意需求)
- 弱點類型(=需求最低的類型)
- 戰鬥狀態(先攻/標準/被襲)
- 視覺線索
- 反應機制

這幾個維度的組合才能撐起多輪驗證的內容密度。
