## 一、核心決策:沒有抽牌

本作不採用任何「抽牌」機制。玩家持有的所有資源完全透明,可隨時掌握使用時機。

### 這個決定的影響
1. **消除運氣不對等**:規劃對了卻因抽牌爛而輸,是 OHK 美學最不能接受的失敗
2. **「擁有」與「使用」分離**:問題不是「我有什麼」而是「我什麼時候用」——這把遊戲從「資訊管理」轉成「決策最佳化」
3. **徹底切斷與 Roguelite 牌組構築的相似度**:玩家用 Into the Breach 思維玩遊戲,不是 Slay the Spire 思維

### 但這也帶來新問題
沒有抽牌,那是什麼**限制**玩家不無腦使用最強的牌?
答案:使用次數制 + 單次消耗的混合機制(見第五章)

## 二、三層卡牌結構

### 第一層:工具牌
| 屬性 | 內容 |
|------|------|
| 效果 | 中低強度、多用途 |
| 限制 | 有使用次數,連戰中消耗 |
| 恢復 | 整備期少量補充 |
| 設計角色 | 日常戰鬥的主力,定義基礎節奏 |

### 第二層:爆發牌
| 屬性 | 內容 |
|------|------|
| 效果 | 高強度、關鍵性 |
| 限制 | 使用後從卡池移除 |
| 恢復 | 無 |
| 設計角色 | 關鍵敵人的解答,每次使用都是大決定 |

### 第三層:角色牌
| 屬性 | 內容 |
|------|------|
| 效果 | 獨特機制、定義角色 |
| 限制 | 單局單次使用 |
| 恢復 | 下一場戰役才重置 |
| 設計角色 | 玩家的「招牌動作」,戰役的記憶點 |

### 三層的設計用意
這個結構自動產生**戰役節奏感**:
- 工具牌定義「日常」——大部分敵人都用工具牌解決,玩家在這個層級磨練基本功
- 爆發牌定義「決斷」——少數關鍵時刻使用,「我認為這個敵人值得我的爆發牌」是重大判斷
- 角色牌定義「故事」——一場戰役只能用一次,會被玩家牢牢記住

OHK 美學的最終呈現不是「我計算贏了」,而是「**我有故事可以講**」。

## 三、第四層:詞條 / 修飾系統(已擱置)

> **此章原始設計在 2026-05-10 結算機制重構後暫時擱置**。詳見 [結算機制演化討論.md](結算機制演化討論.md)。
>
> **擱置原因**:第一次紙上測試驗證詞條觸發率極低(「已揭露」要求前場投情報、「獨擊」與儀式化決戰矛盾),且詞條的乘法計算讓結算負擔過重,違反 OHK「瞬間即儀式」精神。改為類型計數模型後,「次數貢獻」本身已承擔大部分原詞條的策略意義,詞條系統暫時不需要存在。
>
> **未來若想復活**:可能形態為「條件性 +1 次數貢獻」(例如「commit_size==1 時這張卡 +1 衝擊」),但要先確認新動機與測試驗證,不要回到原始的乘法詞條設計。

### 原始設計概要(供參考)

詞條原本設計為「寄生在其他卡上的條件性效果」(獨擊+50% / 已揭露+30% / 本擊含情報+30% 等)。詳細原始內容在 git 歷史中可查。

## 四、卡牌的設計維度

每張卡同時由兩條**正交獨立的主軸** + 三個輔軸定義。

> **注意:本章描述「功能類別」與三個輔軸,構成卡牌的「功能維度」。卡牌另有獨立的「資源類別」維度(工具 / 爆發 / 角色),描述限制機制與稀有度——詳見[第二章三層卡牌結構](#二三層卡牌結構)與[第五章限制機制](#五限制機制)。**
>
> **資源類別**與**功能類別**是兩條獨立的軸,每張卡都同時填寫。例:「反覆可用的揭露卡」=「工具 + 情報收益」;「一次性大揭露」=「爆發 + 情報收益」;「角色專屬的試探招」=「角色 + 複合」。

### 主軸:功能類別

描述卡牌的核心效果——這張卡做什麼:

- **戰鬥收益**:對敵人需求表貢獻次數(主要用於擊殺)
- **情報收益**:揭露 hidden state(弱點類型、各類型需求次數、特殊修飾),讓未來決策更精確
- **複合 / 試探**:輕度次數貢獻 + 輕度揭露,作為「未確認情報時的中間選擇」——讓玩家在「直接擊殺 vs 投入情報」之間有第三條路

### 輔軸一:效率與成本曲線
- 高效率廉價
- 強力高成本
- 條件性爆發
- 遞減效益

### 輔軸二:指向性
- 單一目標
- 範圍 / 多目標
- 限定類型(只能用在某類敵人)
- 自由指定

### 輔軸三:使用節奏
本作為單次 commit 結構,故節奏概念簡化為:
- 立即生效
- 條件觸發
- 持續效果(影響本擊整體)

### 範例:戰鬥收益類的不同設計
| 卡牌 | 規格 | 角色 |
|------|------|------|
| 穩定斬 | 高效廉價 + 單一目標 + 立即 | 基本工具牌 |
| 滿月之刃 | 條件爆發(已揭露+) + 單一 + 立即 | 情報的回報 |
| 連環震盪 | 遞減 + 多目標 + 持續 | 群清的種子 |
| 復仇之擊 | 條件爆發(前場 OHK 失敗) + 限定 + 條件觸發 | 補救機制 |

### 設計原則
**避免「同類型卡感覺都一樣」**——資源池內的同質卡(例如多張「直擊」)是預期的且必要的,但**個體特殊卡**(爆發、角色、複合)需要透過資源類別 + 功能類別的組合形成識別性,讓每張個體卡都有獨特的使用情境。

## 五、限制機制

### 採用混合方式
- **工具牌**:有使用次數限制,整備期少量回補
- **爆發牌**:單次性消耗(用後移除)
- **角色牌**:單局單次(戰役重置)

### 為什麼這個組合好
創造自然的「使用層級感」:
- 工具牌定義「日常」
- 爆發牌定義「決斷」
- 角色牌定義「故事」

每一層都有不同的承諾感,玩家會在不同層級體驗不同類型的爽感。

### 為什麼不選其他機制
- **共用點數池**(Slay the Spire 的能量系統):會讓玩家體驗逐漸接近傳統卡牌遊戲,違背美學
- **互斥綁定**:玩家認知負擔重
- **純單次性消耗**(每張卡整局只能用一次):太極端,需要設計極大的卡池

## 六、整備期的恢復平衡

### 關鍵數字
整備期能回補多少使用次數,**這個單一數字決定遊戲的全部感覺**。

### 平衡指引
- **太多** → 工具牌變「無限資源」,連戰的稀缺感消失,爆發牌變累贅
- **太少** → 玩家被迫過度依賴爆發牌,戰役後期資源枯竭,變計算機題
- **剛剛好** → 工具牌「每場可以用 2–3 張,但永遠不夠完全處理一場戰鬥」

### 建議起點
**整備期回補的使用次數 ≈ 上一輪連戰消耗的 60–70%**

讓玩家永遠在「資源緩慢縮水」的趨勢中,但不至於崩盤。透過特定關卡或事件給予「補充包」作為奇點,創造「這輪我喘到氣了」的鬆緊節奏。

這個比例不是定論,是測試起點。實機調整時會感受到正確的數字。

## 七、「本擊」概念與 Place/Lock 機制

### 定義
本擊 = 玩家在單場戰鬥中,在 commit 階段內提交的卡牌組合。

### 為什麼這個概念重要
本擊是設計上的最小單位,不是「每張卡」:
- 揭露效果依「本擊」內的卡綜合判定
- 結算結果依「本擊」整體類型次數決定
- 玩家的決策對象是「本擊」而非「個別卡」

### 本擊內的兩種卡牌狀態:Place / Lock

本擊雖然是「單一 commit 階段」的決策,內部卡牌有兩個狀態:

| 狀態 | 玩家動作 | 系統行為 | 可逆性 |
|------|---------|---------|--------|
| **Place(出牌)** | 拖卡到本擊區,但未確認 | 不生效、不消耗資源 | ✓ 可隨時撤回/替換 |
| **Lock(鎖定出牌)** | 對單張卡點「鎖定」 | 立即消耗資源 + 觸發 special_effect | ✗ 不可撤回 |

### 為什麼這個機制存在

三個目標:

1. **解決 flexible 分配的 timing 問題**——讓玩家可以在 commit 階段內「先 lock 揭露卡 → 看資訊 → 再分配 flexible」
2. **模擬「物理動作鏈」**——一旦動作開始(lock)就不能收回,但其他動作可以根據資訊調整。配合 OHK 美學的「不可逆」精神
3. **保留單一 commit 階段感受**——所有 lock 仍是同一個本擊的子階段,不是分次戰鬥;玩家做的是「同一決策的子動作」,不是多次 commit

### 流程示意

對未知弱點的敵人,玩家手上有:洞察一擊(揭露當前)、爆裂(+2 flexible)、直擊 ×2

```
[Place 階段] 玩家拖卡到本擊區
   置入:洞察一擊、爆裂、直擊、直擊
   本擊狀態:衝擊 0 / 穿刺 0 / 燃燒 0(flexible 未分配)

[Lock 洞察一擊]
   → 觸發 special_effect:揭露當前敵人弱點 = 衝擊
   → 洞察一擊資源消耗(角色卡用掉)
   → 玩家現在知道弱點

[玩家分配爆裂的 flexible]
   爆裂 +2 → 玩家分配:+2 衝擊

[「結束本擊」按鈕]
   → 鎖定所有未 lock 的 placed 卡(爆裂、直擊×2)
   → 戰鬥層計算
   → 衝擊 = 2(爆裂)+ 1+1(直擊×2)+ 2(洞察貢獻)= 6

[結算]
```

整個流程仍是「**一個本擊**」,只是內部允許資訊→決策的流動。

### 卡牌的 lock_class 三分類(2026-05-11 確立)

並非所有卡都需要 lock。每張卡標記一個 `lock_class`,決定它與 Place/Lock 機制的互動方式:

| lock_class | 行為 | 卡面標示 | 包含卡類 |
|-----------|------|---------|---------|
| **`"none"`(無 lock 需求)** | Lock 與否完全不影響效果。可單獨 lock,也可隨「結束本擊」一起 lock | 卡面**不寫任何 lock 標示** | 純戰鬥卡(資源池 14 張、純戰鬥爆發 3 張、純戰鬥角色 2 張)= **19 張** |
| **`"optional"`(可選 lock)** | Lock 與否**改變揭露對象**:不 lock = 後續廣度;Lock = 當前犧牲廣度換深度 | 卡面寫**兩種模式效果** | 情報卡(偵察、估量、警示)= **3 張** |
| **`"required"`(強制 lock)** | **必須先 lock 才能生效**;「結束本擊」前所有 placed 的 required 卡必須先 lock | 卡面標 🔒 強制 lock 標示 + 分配說明(若是 flexible) | 試探卡 3 張、flexible 爆發/角色 2 張、洞察一擊、全知之眼 = **6 張** |

#### 三類詳細

**a. `lock_class: "none"`(19 張)**
- 玩家拖卡到本擊區即可,不需要單獨 lock
- 「結束本擊」時統一鎖定 + 結算
- 例:直擊、重斬、銳刺、貫穿、點火、預判燒灼、重擊、致命刺、烈焰、殺戮意志、致命凝視

**b. `lock_class: "optional"`(3 張,情報卡)**
- 不 lock(預設模式):結束本擊時觸發「廣度揭露」(此連戰所有剩餘敵人某項資訊)
- Lock(深度模式):lock 時即時揭露當前敵人某項資訊,但放棄後續廣度
- 玩家在「現在 vs 未來」之間做真實的判斷取捨
- 例:偵察(揭露弱點類型)、估量(揭露需求)、警示(揭露精英化)

**c. `lock_class: "required"`(6 張)**
- **必須單獨 lock 才能生效**(只 place 不 lock = 等於沒打出)
- 「結束本擊」按鈕在所有 required 卡都 lock 之前不可用
- flexible 卡 lock 時必須完成類型分配
- 試探卡只作用於當前敵人(動作邏輯)

> **概念區分:情報 vs 試探**
> - **情報卡(b 類)** = 「我完全不知道,想要答案」(揭露具體資訊)
> - **試探卡(c 類)** = 「我有猜測,想驗證」(binary yes/no 確認)
>
> 試探卡只作用當前敵人的設計邏輯:**它是一個動作**,動作不會影響未來的敵人,只能作用於眼前對象。

### Lock 順序的策略意義

對 b 類情報卡:lock 順序變成「**現在揭露 vs 未來揭露**」的明確判斷,不是隱性最優。

對 c 類試探卡 / flexible:必須 lock,但**lock 的時機**仍是策略——
- 早 lock = 先拿揭露資訊 → 後續可調整其他 placed 卡
- 晚 lock = 等其他 lock 完成後再決定

對 a 類純戰鬥卡:lock 與否無差,通常不單獨 lock。

### 規範彙整

| 規則 | 內容 |
|------|------|
| 可逆性 | Place ✓ / Lock ✗ |
| 粒度 | 單張卡 lock,不可批次 |
| 「結束本擊」前置條件 | 所有 placed 的 `required` 卡必須先 lock(否則按鈕禁用) |
| 資源消耗時機 | Lock 時立即消耗(包括資源池的 use 計數) |
| 「結束本擊」動作 | 鎖定所有未 lock 的 placed 卡(僅 a/b 類) + 戰鬥層整體計算 |
| Lock 順序 | 玩家自由決定;順序本身是策略 |
| Lock 過程中新增 placed 卡 | 允許(未影響已 lock 卡) |
| flexible 卡 lock 時 | 必須完成類型分配(必填) |

### special_effect 的觸發時機

依 `lock_class` 而定:

- **`lock_class: "none"`**:special_effect 通常為 null。若有,在「結束本擊」時統一處理(如致命凝視的「對最高需求者 +1」)
- **`lock_class: "optional"`**:雙模式 spec(`spec_unlocked` + `spec_locked`)。不 lock 走 unlocked(在「結束本擊」時觸發);Lock 走 locked(lock 時即時觸發)
- **`lock_class: "required"`**:special_effect 必須在 lock 時觸發才生效

詳見 [卡牌資料結構.md](卡牌資料結構.md) 第三章。

### 設計上的意義

這個機制讓本擊的「最小單位」性質保留(玩家仍是對「本擊整體」做判斷),但同時允許資訊在本擊內動態流動。**「烹飪配方」的概念升級為「動作鏈」**——食材組合的判斷不變,但執行有先後並可隨資訊調整。

### 設計空間延伸(供未來參考)

採用 Place/Lock 機制後保留了幾個未來可擴充的設計空間:

**A. 「強制 lock」作為敵人效果**

未來可能的敵人能力:
- 連動術士:玩家本擊每 lock 一張卡,該敵人「混合任意需求」-1(獎勵承諾)
- 看穿者:玩家若連續 lock 3 張未 reveal 卡,該敵人弱點需求 +1(懲罰盲打)
- 反應型 boss:玩家第一張 lock 動作觸發 boss 揭露自己的隱藏修飾

這些都依賴「lock 是物理動作」這個前提才成立。**保留純戰鬥卡可 lock 的選項就保留了這些設計可能**。

**B. 「未 lock 即未生效」延伸的新卡牌種類**
- 準備型卡牌:Lock 時付出代價,在「結束本擊」時觸發特殊效果
- 連動型卡牌:必須在某張卡 lock 之後才能 lock(時序卡)

這些是**未來擴充的種子**,原型不需要,但設計空間清楚。

## 八、flexible(任意類型)機制

### 定義
某些卡的 contribution 包含 `flexible: N`,代表 N 點次數可由**玩家在 lock 時主動分配**到任一傷害類型。

例:爆裂的 contribution = `{ flexible: 2 }`,玩家在 lock 時可分配為 `+2 衝擊` 或 `+1 衝擊 +1 穿刺` 或 `+2 燃燒` 等。

### 三種被否決的詮釋

設計上曾考慮三種詮釋,最終採用 C(玩家主動分配):

| 詮釋 | 描述 | 為什麼否決 |
|------|------|----------|
| A. 自動最優分配 | 系統自動把 flexible 算給能達標的類型 | 違反「玩家是判斷者」原則 |
| B. 同時對所有類型 +N | flexible 對所有達標路徑都計入 | 太強;弱點對應價值被稀釋 |
| **C. 玩家主動分配** | 玩家在 lock 時必須明確分配到一或多個類型 | ✓ 採用 |

### flexible 在 Place/Lock 機制下的運作

- **Place 階段**:flexible 顯示「未分配」灰色狀態
- **Lock 階段**:玩家必須明確分配(必填),確認後 lock
- 分配可選:
  - 全押一類(如 +2 衝擊)
  - 分散(如 +1 衝擊 +1 穿刺)

### flexible 服務於判斷的方式

flexible 的價值不是「強度」,是「**彈性**」:

1. **未知弱點時的保險**——先 place flexible 卡,等情報卡 lock 後揭露結果,再 lock + 分配
2. **多選擇的策略風格**——分散 vs 集中是玩家的風險偏好
3. **與情報投資的協同**——揭露 → 知道弱點 → flexible 全押弱點 = 最大化情報回報

### 為什麼不採用「自動分配」

如果系統自動把 flexible 算給弱點:
- 玩家不需要思考分配
- 但**「玩家是判斷者」原則被違反**——系統替玩家做最佳化決策
- flexible 變成「免費 +N 弱點類型」這種無腦增益,失去判斷張力
- 違反 [呈現層原則](設計概念.md#十二呈現層原則)「系統不替玩家做計算決策」

採用 C(玩家主動分配)保留:
- 「我要怎麼分配」的判斷空間
- 不確定下的「先擱著等情報」彈性
- 多目標 commit 的擴充潛力(未來如果有多敵人 commit)

### flexible 卡的真實 niche

| 卡 | flexible 部分 | 玩家在不同情境的選擇 |
|----|-------------|----------------|
| 爆裂(爆發 +2 flex) | +2 任一 | 不確定弱點時:place 等揭露;確定弱點時:直接 lock+分配 |
| 試探擊(複合 +1 flex) | +1 任一 | 通常先 lock 拿揭露 + 分配 |
| 洞察一擊(角色 +2 flex) | +2 任一 | 通常先 lock 揭露當前弱點 + 立即分配對應 |

## 九、卡牌獲得的隨機性

### 採用方式
每局獲得不同卡牌組合,但**玩家看見並選擇**,不做盲抽。

### 實作建議
- 整備期提供 3–4 張候選卡,玩家挑 1 張
- 卡片屬性變化時讓玩家看見完整資訊再選擇
- 不允許「隨機附加效果」這種失控變化

### 卡片變異的可能形態(類型計數模型下)
同樣的個體卡可能出現次數貢獻變化(例如某張爆發卡這次給 +3 衝擊,下次給 +2 衝擊 +1 穿刺),或同類卡牌池可能臨時增加類型張數(這局衝擊池 5 張、下局穿刺池 6 張)。讓玩家每局重新評估資源分布。

## 十、多角色設計

### 角色不只是一張角色牌的差異
要讓多角色有意義,每個角色不能只差一張角色牌——他們應該有**整體玩法傾向的不同**。

每個角色實際上是「**一套不同類型偏好的資源池 + 數張個體角色卡**」。差異可從:資源池中各類型張數比例、角色卡的特殊機制、爆發卡組合三方面建立。

### 角色範例
- **狩獵者**:資源池偏穿刺與情報,角色卡擅長「揭露後一擊」,鼓勵先偵察再擊殺
- **狂戰士**:資源池偏衝擊與通用,角色卡擅長「無視類型修正」,鼓勵直接賭博
- **學者**:資源池偏複合卡,角色卡擅長「跨場連鎖揭露」,鼓勵情報累積

### 開發建議:漸進式內容擴張
**先做 1 個角色把核心打磨完,再考慮第二個角色作為第一個正式更新**——這是最健康的開發路徑。如果一開始就規劃 5 個角色,每個都會被攤薄。

對單人開發或小團隊尤其重要。

## 十一、原型卡牌分配計畫(v2,類型計數模型)

本章描述原型階段(2 組連戰、5 隻敵人、約 30 張卡)的卡牌分配,對應 [類型計數結算模型](設計概念.md#十一單一檢查原則)。

> **v1 (倍率模型) 的舊分配計畫已在第一次紙上測試後棄用**,詳見 [結算機制演化討論.md](結算機制演化討論.md)。

### 設計判準

從 [敵人模板](敵人模板.md) 的 5 個原型敵人需求表反推卡牌需求(具體數值待敵人模板更新後確認):

| 敵人模板 | 弱點類型 | 戰鬥需求估算 |
|---------|---------|------------|
| 哨兵 | 衝擊或穿刺 | 弱點 2 張 ★ 即過 |
| 巡守 | 衝擊/穿刺/燃燒 | 弱點 2-3 張 ★★ |
| 重甲衛兵 | 衝擊或穿刺 | 弱點 3 張 ★★ |
| 伏擊者 | 穿刺或燃燒 | 弱點 2 張 ★★ |
| 領主 | 衝擊或穿刺 | 弱點 3-4 張,需爆發/角色補強 |

弱點類型出現次數:穿刺 5 次 / 衝擊 4 次 / 燃燒 2 次

### 卡池結構(類型計數模型)

採用「**資源池(同質) + 個體特殊卡**」混合結構:

#### Layer A:傷害資源池(同質,可重複)

| 池 | 強度 | 數量 | 每張貢獻 | 池總供給 |
|----|------|------|---------|---------|
| 衝擊 ★ 池(直擊) | ★ | 4 張 | +1 衝擊 | +4 衝擊 |
| 衝擊 ★★ 池(重斬) | ★★ | 2 張 | +2 衝擊 | +4 衝擊 |
| 穿刺 ★ 池(銳刺) | ★ | 4 張 | +1 穿刺 | +4 穿刺 |
| 穿刺 ★★ 池(貫穿) | ★★ | 2 張 | +2 穿刺 | +4 穿刺 |
| 燃燒 ★ 池(點火) | ★ | 3 張 | +1 燃燒 | +3 燃燒 |
| 燃燒 ★★ 池(預判燒灼) | ★★ | 2 張 | +2 燃燒 | +4 燃燒 |
| **小計** | | **17 張** | | |

弱點類型總供給比例與敵人需求對齊:衝擊 8 / 穿刺 8 / 燃燒 7 = 約 1:1:0.9。

#### Layer B:個體特殊卡(獨特效果)

| 類別 | 數量 | 範例(實際規格待重設計) |
|------|------|--------------------|
| 情報卡(揭露下一場單一資訊) | 3 張 | 揭弱點類型、揭最低需求次數、揭精英化 |
| 試探卡(輕度貢獻+揭露) | 3 張 | +1 衝擊揭露弱點、+1 穿刺揭最低需求、+1 燃燒揭精英化 |
| 爆發卡(單次性) | 4 張 | +3 衝擊、+3 穿刺、+3 燃燒、+2 任一(玩家分配) |
| 角色卡(戰役單次) | 4 張 | 招牌大招、跨場揭露、複合貢獻等 |
| **小計** | **14 張** | |

#### 總計:17 + 14 = 31 張(可微調至 30)

### 強度層級的新意義

- ★ = +1 次數貢獻
- ★★ = +2 次數貢獻
- ★★★ = +3 次數貢獻(僅爆發/角色)
- 「★★★ 工具」不存在(設計上若需要 +3 級的可重複卡,直接升級為爆發)

### 對 5 個敵人的擊殺路徑驗算(待敵人需求表更新後確認)

驗算原則:**任一類型達到該類型需求即 OHK**。
- 弱點對應 = 走最低需求路徑(最省)
- 不對應 = 用其他類型達到該類型較高需求(較貴)
- 走「混合任意」路徑 = 不限類型湊總次數(最貴,通常 ≥ 弱點需求 ×3)

具體驗算將在敵人模板與卡牌規格更新後進行。

## 十二、待解決的設計問題

1. **手牌規模**:玩家持有的卡片總量上限(6–10 張範圍待測試)
2. **本擊規模**:單次 commit 的卡片數量上限
3. **詞條系統復活的可能性**:目前已擱置,未來若加回需以「條件性 +1 次數貢獻」形態而非乘法
4. **使用次數回補速率**:整備期回補多少,需大量實測調整
5. **跳過卡 / 情境收益類**:種類、限制、銜接後續戰鬥的方式;原型暫不實作,可能歸類於角色特殊卡類別(作為第四種功能類別「情境收益」)待後續研究
6. **角色牌的具體效果範圍**:多強才合適?多獨特才有記憶點?

卡牌系統的骨架已經明確。下一步邏輯上應該處理:

**敵人設計** — 既然玩家手段已定,現在要設計能讓這些手段都有舞台的敵人。
