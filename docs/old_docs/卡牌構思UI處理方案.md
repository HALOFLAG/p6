# 卡牌構思:UI 處理方案

## 文件目的與狀態

本文件記錄關於「開局太多卡」問題的解決方案討論,確認以 **UI 工具 + 教學期 UI 預設**為主軸的處理方向。

**結論:採用 UI 解法**,不採用「教學戰役卡池與主戰役卡池分離」的內容方案。30 張卡牌保持統一設計,差別只在 UI 預設。

**前置背景**:本問題源自 [戰役章節式架構討論](戰役章節式架構討論.md) 中被擱置的原始痛點——章節制提案被擱置後,「開局太多卡」仍需獨立處理。

---

## 一、問題的真正樣貌

「開局太多卡」其實是三個不同層次的子問題疊加:

1. **卡牌絕對數量**:手上有 N 張,眼睛要掃描多少張
2. **每張卡的複雜度**:特殊效果讓部分個體卡看起來「會做很多事」(原模型有詞條,但已擱置)
3. **選擇空間**:C(N, k) 的組合爆炸——10 張選 4 張就有 210 種組合

這三層會獨立失控,解法必須對應到具體哪一層在出問題。

**一個重要觀察**:選擇空間的問題很大一部分根源不在卡,而在玩家**還沒學會讀敵人**。如果玩家能從敵人視覺暗示直接讀出弱點,他看 10 張卡的選擇就自然收斂到「有哪些對應類型」——選擇空間從 210 壓縮到個位數。

也就是說,**教學戰首先要教的是「讀敵人」而非「用卡」**。卡牌的選擇困難會隨著「讀敵人能力」的提升自動消解。

---

## 二、解法的兩條路線

### 路線一:內容解法(已評估,未採用)

例如:教學戰役發少量基礎卡、主戰役才完整展開。

**為什麼擱置**:
- 需要為「教學專用」設計獨立卡池,內容工作量增加
- 教學戰到主戰之間有「相位變化」(卡突然多了)
- 對個人開發者複雜度高

### 路線二:UI 解法(採用)

所有卡牌一視同仁設計,「太多卡」感受透過 UI 工具與教學期 UI 預設處理。

**為什麼採用**:
- 30 張卡保持統一設計,不需區分教學/主戰
- 無相位變化(內容不變,只是 UI 變開放)
- 老手體驗不受影響(UI 輔助可關閉)
- UI 工具本身是學習素材(玩家學會「整理資訊」的元能力)

---

## 三、UI 解法的核心區分:A 類 vs B 類

UI 解法本身分兩種,**這個區分是整個方案的關鍵**:

| 路線 | 描述 | 是否符合 P6 原則 |
|------|------|---------------|
| **A 類:被動工具** | UI 提供整理工具,玩家自己決定怎麼用 | ✓ 系統幫玩家整理已知資訊 |
| **B 類:主動推薦** | UI 分析當下情境,系統替玩家建議 | ✗ 系統替玩家做判斷 |

### 具體舉例

- 「**這個類別的敵人推薦某類卡到前列**」→ B 類,**必須拒絕**
  - 系統知道敵人弱點(它有 hidden state),依此推薦 = 系統洩露 hidden state
  - 等同 UI 偷偷告訴玩家「弱點是衝擊」,讓「猜弱點」「投情報換確定性」整套設計變得多餘
  - 同時破壞「老手能做到純看視覺暗示擊殺」的設計合約——因為新手也會被 UI 暗示

- 「**簡易篩選機制讓玩家自己抓需要類型**」→ A 類,**可採用**
  - 玩家自己決定「我要找衝擊類」,UI 只是幫他過濾
  - 沒有 hidden state 洩露,純粹減少眼睛掃描成本

### 區分原則

> **系統可以整理資訊,但不能做判斷。**

這跟呈現層原則(設計概念十二章)的「系統是計算者,玩家是判斷者」完全一致——只是把「計算」延伸到「整理」。

---

## 四、可採用的 A 類 UI 工具集

| 工具 | 功能 | 適用情境 |
|------|------|---------|
| **類型篩選按鈕** | 衝擊 / 穿刺 / 燃燒 / 通用 / 情報 切換顯示 | 玩家自己選類型 |
| **玩家自定義釘選** | 把常用卡釘到前列 | 老手建立慣用組合 |
| **本擊建構區與庫存區分離** | 拖拉到「本擊」槽,庫存看著清楚 | 視覺上分離決策區與資源區 |
| **Place vs Lock 兩區設計** | 本擊區再細分為「未鎖定 placed」與「已鎖定 locked」兩子區 | 對應 Place/Lock 機制 |
| **flexible 分配滑桿** | flexible 卡 lock 時跳出分配介面 | 對應 flexible 機制 |
| **緊湊/展開切換** | 預設緊湊呈現(類型+次數貢獻+強度層級),點擊展開詳細(特殊效果) | 降低初始視覺密度 |
| **本擊範本儲存** | 玩家儲存「常用組合」可一鍵套用 | 老手提速 |

這些都是**讓玩家更輕鬆地操作他已經知道要做的事**,不是替他做判斷。

### Place / Lock UI 細節

詳細機制定義見 [卡牌構思.md 第七章 Place/Lock 機制](卡牌構思.md)。UI 層面要點:

**本擊區雙子區設計**

```
┌─────────────────────────────────────┐
│ 本擊區                              │
│ ┌─────────────┐  ┌──────────────┐ │
│ │ Place(可編輯)│  │ Lock(已鎖定) │ │
│ │ - 卡 A      │  │ - 卡 X (✗ 不可動)│ │
│ │ - 卡 B      │  │              │ │
│ └─────────────┘  └──────────────┘ │
│                                    │
│ 即時本擊狀態:                       │
│ 衝擊 +X / 穿刺 +Y / 燃燒 +Z       │
│                                    │
│ [結束本擊] 按鈕                     │
└─────────────────────────────────────┘
```

**Lock 操作流程**:
1. Place 區的卡點「lock」按鈕(或拖到 lock 區域)
2. 系統 confirm「鎖定後將立即消耗資源並觸發效果,確認?」
3. 確認後卡進入 lock 區,special_effect 立即生效
4. Lock 區的卡視覺上「定格」(灰底、邊框加粗、無法拖動)

**flexible 分配 UI(lock 時必填)**:

```
玩家將「爆裂(+2 任一)」拖到本擊區
玩家點擊「lock」按鈕
↓
系統跳出分配介面:
  必須分配總和 = 2
  衝擊 [○○]  穿刺 [○○]  燃燒 [○○]
  玩家拖動分配,例如 衝擊 +2
↓
確認 → 卡片進入 lock 區並標註「+2 衝擊」
```

### 三類卡牌的卡面設計差異

對應 [卡牌構思.md 第七章 lock_class 三分類](卡牌構思.md):

#### 類 a:`"none"`(19 張純戰鬥卡)
- 卡面**不寫任何 lock 相關文字**
- 純粹顯示:名稱、類別、貢獻、強度、使用次數
- 玩家對待方式:直接拖到本擊區,「結束本擊」一起鎖

#### 類 b:`"optional"`(3 張情報卡)
- 卡面分為**兩段效果**:
  ```
  [偵察] 情報卡 ★
  使用次數: ●●

  ❒ 不 lock(預設):
    結束本擊時揭露此連戰所有
    剩餘敵人的弱點類型
  
  ❑ Lock 啟動:
    即時揭露當前敵人弱點類型
  ```
- 視覺強調「兩種模式」可選
- 玩家對待方式:看情境決定要 lock(深度)還是不 lock(廣度)

#### 類 c:`"required"`(6 張)
- 卡面標**🔒 強制 Lock 標示**
- 寫清楚「使用時須 lock」+「flexible 卡需分配類型」
- 範例:
  ```
  [試探斬] 試探卡 ★ 🔒
  +1 衝擊
  使用次數: ●●●

  🔒 Lock 啟動(必須):
    揭露當前敵人弱點是否
    為衝擊(yes/no)
  ```
  
  flexible 卡額外標示:
  ```
  [爆裂] 爆發卡 ★★ 🔒
  +2 任一(需分配)
  使用次數: ●

  🔒 Lock 啟動(必須):
    Lock 時須選擇傷害類型
    分配 +2(可全押或分散)
  ```

### 「結束本擊」按鈕的狀態邏輯

UI 層面:
- 預設狀態:可用(允許玩家直接結束)
- **若有 placed 但未 lock 的 c 類卡 → 按鈕禁用 + 提示**(「有未 lock 的強制卡」)
- 玩家必須先 lock 完所有 c 類卡,才能結束本擊

## 五、準 B 類的中間方案(已採用)

有一種看起來像 B 類但實際合規的做法:

### 「基於玩家已揭露資訊的排序輔助」

如果玩家花了情報牌揭露「弱點是衝擊」,UI 把衝擊類卡推到前列。

**為什麼合規**:
- 這個資訊已經是玩家知道的(他剛揭露的)
- UI 只是「幫他不要忘記」,不是「替他發現」
- 沒有任何 hidden state 被洩露

**與 B 類的差別**:B 類是「依 hidden state 推薦」(系統知道但玩家不知道的東西),這個方案是「依玩家已知資訊整理」(雙方都知道的東西)。

## 六、教學期 UI 的特殊處理

UI 解法相對於內容解法的真正優勢——教學期可以由**設計者預先指定 UI 預設**,把「教學」內建到 UI 介面裡。

### 採用「強引導」路線

教學期應該強烈引導,但這個強引導的存在感本身要明顯到玩家知道「啊我在被引導,等我熟了就關掉」。弱引導(微亮邊框)容易模糊不清,反而讓玩家不確定這是什麼。

### 教學節奏範例

| 戰鬥 | UI 預設 | 配套 |
|------|--------|------|
| 教學戰 1(哨兵) | 灰階所有不相關卡,只剩 6 張(衝擊+穿刺各 3 張) | NPC 對話「先看這幾張,想想哪一張對應它的弱點」 |
| 教學戰 2-3 | 逐步解鎖 UI 工具(篩選按鈕、釘選、範本) | 玩家學會「自己整理資訊」的技能 |
| 主戰役起 | UI 完全開放,所有工具可用 | 玩家自由運用 |

### 跟 B 類的差異

教學期 UI 的「相關判斷」不是執行時根據敵人推測,而是**設計者預先指定**——就像教科書第一頁刻意只放 3 個例子,不是書本「分析學生需要什麼」推薦。設計者在做課程設計,不是執行時判斷。

---

## 七、對卡牌設計的影響

走 UI 解法之後,卡牌設計反而**更自由**:

| 對比項 | 內容解法 | UI 解法(採用) |
|-------|---------|--------------|
| 30 張卡的設計 | 需區分教學卡(精簡基礎)與主戰卡(完整) | 30 張全部一視同仁,個體特殊效果從第一張就可以有 |
| 玩家進階模型 | 「進入新階段拿到新卡」 | 「逐步學會用 UI 工具 + 逐步見識更多卡」 |
| 教學到主戰的銜接 | 有相位變化(卡突然多)需處理 | 無相位變化,內容不變只有 UI 變開放 |
| 重玩體驗 | 重玩時教學戰仍是教學戰 | 重玩時可以一開始就關閉所有 UI 輔助 |

**整體效應**:卡牌設計可以以「整體一致」為前提推進,不需要為了教學生出兩套系統。

## 八、實作成本與工作項

採用此方案需實作的 UI 元件(對個人開發者+AI 輔助寫程式可控,但需明確規範):

### A 類 UI 工具
- 類型篩選按鈕(小)
- 玩家自定義釘選(小)
- 緊湊/展開切換(小)
- 本擊建構區與庫存區分離(中)
- 本擊範本儲存(中,需序列化機制)

### 準 B 類中間方案
- 已揭露資訊的排序輔助邏輯(中,需追蹤每場揭露狀態)

### 教學期專屬
- UI 工具逐步解鎖的引導流程(中,需設計教學節奏)
- 教學期的「預設突顯」配置(小,但需與每場戰鬥的教學重點對應)
- 灰階/強調的視覺處理(小)

## 九、已確認的決策

1. ✓ **區分 A 類 vs B 類**:B 類「依敵人類型推薦卡」明確排除
2. ✓ **採用準 B 類中間方案**:基於玩家已揭露資訊的排序輔助
3. ✓ **教學期 UI 走「強引導」路線**:灰階不相關卡,明確讓玩家知道「我被引導」
