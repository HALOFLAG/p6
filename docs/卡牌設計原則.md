## 一、核心哲學:沒有抽牌

P6 **不採用任何「抽牌」機制**。玩家持有的所有資源完全透明,可隨時掌握使用時機。

### 這個決定的影響

1. **消除運氣不對等**:規劃對了卻因抽牌爛而輸,是 OHK 美學最不能接受的失敗
2. **「擁有」與「使用」分離**:問題不是「我有什麼」而是「我什麼時候用」——這把遊戲從「資訊管理」轉成「決策最佳化」
3. **徹底切斷與 Roguelite 牌組構築的相似度**:玩家用 Into the Breach 思維玩遊戲,不是 Slay the Spire 思維

### 但這也帶來新問題

沒有抽牌,那是什麼**限制**玩家不無腦使用最強的牌?

→ 答案:**使用次數制 + 單次消耗的混合機制**(見第二章「三層卡牌結構」)

## 二、三層卡牌結構:工具 / 爆發 / 角色

### 第一層:工具牌

| 屬性 | 內容 |
|------|------|
| 效果 | 中低強度、多用途 |
| 限制 | 有使用次數,連戰中消耗 |
| 恢復 | 整備期少量補充 |
| 設計角色 | 日常戰鬥的主力,定義基礎節奏 |

### 第二層:爆發牌

| 屬性 | 內容 |
|------|------|
| 效果 | 高強度、關鍵性 |
| 限制 | 使用後從卡池移除 |
| 恢復 | 無 |
| 設計角色 | 關鍵敵人的解答,每次使用都是大決定 |

### 第三層:角色牌

| 屬性 | 內容 |
|------|------|
| 效果 | 獨特機制、定義角色 |
| 限制 | 單局單次使用 |
| 恢復 | 下一場戰役才重置 |
| 設計角色 | 玩家的「招牌動作」,戰役的記憶點 |

### 三層的設計用意

這個結構自動產生**戰役節奏感**:
- 工具牌定義「日常」——大部分敵人都用工具牌解決,玩家在這個層級磨練基本功
- 爆發牌定義「決斷」——少數關鍵時刻使用,「我認為這個敵人值得我的爆發牌」是重大判斷
- 角色牌定義「故事」——一場戰役只能用一次,會被玩家牢牢記住

> OHK 美學的最終呈現不是「我計算贏了」,而是「**我有故事可以講**」。

### 為什麼不選其他機制

- **共用點數池**(Slay the Spire 的能量系統):會讓玩家體驗逐漸接近傳統卡牌遊戲,違背美學
- **互斥綁定**:玩家認知負擔重
- **純單次性消耗**(每張卡整局只能用一次):太極端,需要設計極大的卡池

## 三、兩軸正交分類:資源類別 × 功能類別

每張卡同時由**兩條正交獨立的主軸**定義,每張卡都同時填寫兩個軸。

### 軸一:資源類別(描述限制機制與稀有度)

| 類別 | 限制機制 | 設計定位 |
|------|---------|---------|
| 工具 | 有使用次數限制(整備期回補) | 日常戰鬥主力 |
| 爆發 | 單次性消耗(用後移除) | 關鍵時刻決斷 |
| 角色 | 戰役單次(戰役重置) | 招牌動作 / 故事性 |

### 軸二:功能類別(描述卡的核心效果)

| 類別 | 主要功能 | 觸發時機 |
|------|---------|---------|
| 戰鬥收益 | 對敵人需求表貢獻次數 | 結束本擊時整體結算 |
| 情報收益 | 揭露 hidden state | Lock 時觸發 |
| 複合 / 試探 | 輕度次數貢獻 + 輕度揭露 | Lock 時觸發揭露,結束本擊時計算貢獻 |

### 二維展開後的卡牌空間

|  | 戰鬥收益 | 情報收益 | 複合 |
|---|---------|---------|------|
| **工具** | 反覆可用的戰鬥卡(資源池形態) | 反覆可用的揭露卡(個體) | 反覆可用的試探卡(個體) |
| **爆發** | 一次性高貢獻(個體) | 一次性大揭露(個體) | 高效試探(個體) |
| **角色** | 招牌大招(個體) | 角色專屬揭露(個體) | 角色專屬複合(個體) |

### 範例

- 「反覆可用的揭露卡」=「工具 + 情報」=偵察、估量、警示
- 「一次性大揭露」=「角色 + 情報」=全知之眼
- 「角色專屬的試探招」=「角色 + 複合」=洞察一擊

### 設計原則:避免「同類型卡感覺都一樣」

資源池內的同質卡(例如多張「直擊」)是預期的且必要的,但**個體特殊卡**(爆發、角色、複合)需要透過資源類別 + 功能類別的組合形成識別性,讓每張個體卡都有獨特的使用情境。

### 待研究:第四個功能類別「情境收益」

跳過卡 / 迴避卡這類「不打也不查,直接跳過」的功能。原型不實作,可能歸類於角色特殊卡類別,待後續研究。

## 四、卡牌形態:資源池卡 vs 個體卡

### 兩種形態的區分

- **資源池卡**(同質卡):多張完全相同的卡共用同一個 template,只在 `工具 × 戰鬥收益` 象限存在(原型階段)
- **個體卡**(獨特卡):每張卡有獨特設計,涵蓋其餘 8 個象限(以及工具×戰鬥的少數例外)

### 為什麼有資源池卡

讓玩家專注於「投幾張」的決策而非「選哪張」的記憶負擔。

### 為什麼有個體卡

保留「招牌動作」「特殊效果」「情報投資」等需要獨特性的設計。

### 強度層級

- ★ = +1 次數貢獻(工具池基本)
- ★★ = +2 次數貢獻(工具池進階)
- ★★★ = +3 次數貢獻(僅爆發/角色)
- 「★★★ 工具」不存在(設計上若需要 +3 級的可重複卡,直接升級為爆發)

## 五、lock_class 三分類:none / optional / required

每張卡標記一個 `lock_class`,決定它與 Place/Lock 機制的互動方式。

> Place/Lock 機制完整定義見 [遊戲核心系統機制.md 第一章](遊戲核心系統機制.md)。

### 三類定義

| lock_class | 行為 | 卡面標示 | 包含卡類 |
|-----------|------|---------|---------|
| **`"none"`(無 lock 需求)** | Lock 與否完全不影響效果 | 卡面**不寫任何 lock 標示** | 純戰鬥卡(資源池 14 張、純戰鬥爆發 3 張、純戰鬥角色 2 張)= **19 張** |
| **`"optional"`(可選 lock)** | Lock 與否**改變揭露對象**:不 lock = 後續廣度;Lock = 當前犧牲廣度換深度 | 卡面寫**兩種模式效果** | 情報卡(偵察、估量、警示)= **3 張** |
| **`"required"`(強制 lock)** | **必須先 lock 才能生效** | 卡面標 🔒 強制 lock 標示 + 分配說明(若是 flexible) | 試探卡 3 張、flexible 爆發/角色 2 張、洞察一擊、全知之眼 = **6 張** |

### 三類詳細

**a. `lock_class: "none"`(19 張)**
- 玩家拖卡到本擊區即可,不需要單獨 lock
- 「結束本擊」時統一鎖定 + 結算
- 例:直擊、重斬、銳刺、貫穿、點火、預判燒灼、重擊、致命刺、烈焰、殺戮意志、致命凝視

**b. `lock_class: "optional"`(3 張,情報卡)**
- 不 lock(預設模式):結束本擊時觸發「廣度揭露」(此連戰所有剩餘敵人某項資訊)
- Lock(深度模式):lock 時即時揭露當前敵人某項資訊,但放棄後續廣度
- 玩家在「現在 vs 未來」之間做真實的判斷取捨
- 例:偵察(揭露弱點類型)、估量(揭露需求)、警示(揭露敵人類別)

**c. `lock_class: "required"`(6 張)**
- **必須單獨 lock 才能生效**(只 place 不 lock = 等於沒打出)
- 「結束本擊」按鈕在所有 required 卡都 lock 之前不可用
- flexible 卡 lock 時必須完成類型分配
- 試探卡只作用於當前敵人(動作邏輯)

### 概念區分:情報 vs 試探

- **情報卡(b 類)** = 「我完全不知道,想要答案」(揭露具體資訊)
- **試探卡(c 類)** = 「我有猜測,想驗證」(binary yes/no 確認)

試探卡只作用當前敵人的設計邏輯:**它是一個動作**,動作不會影響未來的敵人,只能作用於眼前對象。

### Lock 順序的策略意義

對 b 類情報卡:lock 順序變成「**現在揭露 vs 未來揭露**」的明確判斷,不是隱性最優。

對 c 類試探卡 / flexible:必須 lock,但**lock 的時機**仍是策略——
- 早 lock = 先拿揭露資訊 → 後續可調整其他 placed 卡
- 晚 lock = 等其他 lock 完成後再決定

對 a 類純戰鬥卡:lock 與否無差,通常不單獨 lock。

## 六、contribution 機制與 flexible

### contribution 欄位的詳細說明

`contribution` 是一個 dict,描述本卡對各類型的次數貢獻:

```json
// 純衝擊卡
{ "impact": 1 }

// 強衝擊卡
{ "impact": 2 }

// 複合卡(多類型)
{ "impact": 1, "pierce": 1 }

// 通用 / 玩家可選類型
{ "flexible": 2 }

// 純情報卡(不貢獻次數)
null
```

### flexible 的定義

某些卡的 contribution 包含 `flexible: N`,代表 N 點次數可由**玩家在 lock 時主動分配**到任一傷害類型。

例:爆裂的 contribution = `{ flexible: 2 }`,玩家在 lock 時可分配為 `+2 衝擊` 或 `+1 衝擊 +1 穿刺` 或 `+2 燃燒` 等。

### flexible 在 Place/Lock 機制下的運作

- **Place 階段**:flexible 顯示「未分配」灰色狀態
- **Lock 階段**:玩家必須明確分配(必填),確認後 lock
- 分配可選:
  - 全押一類(如 +2 衝擊)
  - 分散(如 +1 衝擊 +1 穿刺)

### 三種詮釋,採用 C(玩家主動分配)

| 詮釋 | 描述 | 為什麼否決 |
|------|------|----------|
| A. 自動最優分配 | 系統自動把 flexible 算給能達標的類型 | 違反「玩家是判斷者」原則 |
| B. 同時對所有類型 +N | flexible 對所有達標路徑都計入 | 太強;弱點對應價值被稀釋 |
| **C. 玩家主動分配** | 玩家在 lock 時必須明確分配到一或多個類型 | ✓ 採用 |

### flexible 服務於判斷的方式

flexible 的價值不是「強度」,是「**彈性**」:

1. **未知弱點時的保險**——先 place flexible 卡,等情報卡 lock 後揭露結果,再 lock + 分配
2. **多選擇的策略風格**——分散 vs 集中是玩家的風險偏好
3. **與情報投資的協同**——揭露 → 知道弱點 → flexible 全押弱點 = 最大化情報回報

## 七、卡牌資料欄位完整定義

卡牌資料分三層:

- **卡牌定義**(Card Definition):靜態資料,卡片本身的設計屬性,跨戰役不變
- **卡牌實例池**(Card Instance Pool):戰役進行中的可用張數狀態(資源池卡)
- **卡牌實例狀態**(Card Instance State):戰役進行中的個別卡片狀態(剩餘使用次數等)

### 卡牌定義(靜態資料)

| 欄位名 | 型別 | 必填 | 用途 | 範例 |
|-------|------|------|------|------|
| `id` | string | ✓ | 系統內部唯一識別 | `"tool_pool_impact_basic"` |
| `name` | string | ✓ | 玩家識別 | `"直擊"` |
| `resource_class` | enum | ✓ | 資源類別軸 | `"tool"` / `"burst"` / `"character"` |
| `function_class` | enum | ✓ | 功能類別軸 | `"combat"` / `"intel"` / `"compound"` |
| `card_form` | enum | ✓ | 卡牌形態 | `"pool"`(資源池) / `"individual"`(個體) |
| `weakness_type` | enum | ✓ | 該卡所屬的弱點類型 | `"impact"` / `"pierce"` / `"burn"` / `"generic"` / `"none"` |
| `contribution` | object \| null | ✓ | 該卡對戰鬥層的次數貢獻 | `{ impact: 2 }` 或 `{ flexible: 2 }` 或 `null` |
| `strength_level` | enum | ✓ | 玩家定性顯示 | `1` / `2` / `3`(對應 ★ ★★ ★★★) |
| `pool_size` | number \| null | △ | 資源池形態時:該池的張數;個體形態時為 null | `3`(直擊池有 3 張) |
| `use_limit` | number | ✓ | 每張卡的使用次數上限 | `3`(★)/ `2`(★★)/ `1`(爆發/角色) |
| `lock_class` | enum | ✓ | Lock 行為類別 | `"none"` / `"optional"` / `"required"` |
| `special_effect` | object \| null | ✓ | 特殊效果(揭露/影響其他卡等) | 見第八章 |
| `description` | string | ✓ | 玩家定性簡述 | `"對單一目標貢獻 +1 衝擊"` |

### 卡牌實例池(資源池形態,動態狀態)

| 欄位名 | 型別 | 範圍 | 用途 |
|-------|------|------|------|
| `card_id` | string | — | 對應卡牌定義 |
| `pool_remaining` | number | 0 ~ pool_size × use_limit | 該池中所有卡牌的剩餘使用次數總和 |

> 簡化模型:不追蹤池內每張卡的個別狀態,只追蹤總可用次數。

### 卡牌實例狀態(個體卡,動態狀態)

| 欄位名 | 型別 | 範圍 | 用途 |
|-------|------|------|------|
| `card_id` | string | — | 對應卡牌定義 |
| `remaining_uses` | number | 0 ~ use_limit | 該卡剩餘使用次數 |
| `available` | boolean | — | 該卡當前是否仍在玩家手上(爆發牌用後 = false 移除) |

## 八、special_effect 結構與 action 列表

`special_effect` 用於非戰鬥層的效果(揭露、影響其他卡計算等)。

### 觸發時機與 lock_class 的關係

- **`lock_class: "none"`**:special_effect 通常為 null。若有,在「結束本擊」時統一處理(如致命凝視的「對最高需求者 +1」)
- **`lock_class: "optional"`**:special_effect 必須有 **`spec_unlocked` + `spec_locked` 雙模式**
- **`lock_class: "required"`**:special_effect 必須在 lock 時觸發才生效

### 結構

**lock_class 為 "none" 或 "required" 時(單模式)**

| 子欄位 | 型別 | 用途 |
|-------|------|------|
| `spec` | object | 機器規格(action + 參數) |
| `display_text` | string | 玩家看到的人話描述 |

**lock_class 為 "optional" 時(雙模式)**

| 子欄位 | 型別 | 用途 |
|-------|------|------|
| `spec_unlocked` | object | 不 lock 時的機器規格 |
| `display_text_unlocked` | string | 不 lock 時的描述 |
| `spec_locked` | object | Lock 時的機器規格 |
| `display_text_locked` | string | Lock 時的描述 |

### action 列表(2026-05-11 更新)

| action | scope/target | 說明 | 使用於 |
|--------|--------------|------|--------|
| `reveal_remaining_weakness_types` | `current_battle_chain` | 揭露此連戰剩餘敵人的弱點類型 | 偵察(不 lock) |
| `reveal_weakness_type` | `current` | 揭露當前敵人弱點類型 | 偵察(lock)、洞察一擊 |
| `reveal_remaining_min_requirements` | `current_battle_chain` | 揭露此連戰剩餘敵人的最低需求次數 | 估量(不 lock) |
| `reveal_full_requirements` | `current` | 揭露當前敵人完整需求表 | 估量(lock) |
| `reveal_remaining_enemy_class` | `current_battle_chain` | 揭露此連戰剩餘敵人的類別(兔子/狼/豹/熊) | 警示(不 lock) |
| `reveal_enemy_class` | `current` | 揭露當前敵人類別 | 警示(lock) |
| `reveal_binary` + `subject: is_weakness_impact` | `current` | 二元揭露:弱點是否為衝擊 | 試探斬 |
| `reveal_binary` + `subject: is_min_req_low` | `current` | 二元揭露:最低需求是否 ≤ 2 | 探刺 |
| `reveal_binary` + `subject: is_failure_game_over` | `current` | 二元揭露:失敗是否會 GAME OVER | 試探擊 |
| `reveal_all_remaining_full` | `current_battle_chain` + includes 陣列 | 揭露剩餘所有敵人完整資訊 | 全知之眼 |
| `reveal_current_weakness` | `current` | 揭露當前敵人弱點(影響本擊內 flexible 卡分配) | 洞察一擊 |
| `bonus_against_highest` | — | 對連戰最高需求者所有貢獻 +1 | 致命凝視 |

### 範例

**lock_class: "optional" — 偵察的雙模式**

```json
{
  "spec_unlocked": { "action": "reveal_remaining_weakness_types", "scope": "current_battle_chain" },
  "display_text_unlocked": "不 lock:結束本擊時揭露此連戰所有剩餘敵人的弱點類型",
  "spec_locked": { "action": "reveal_weakness_type", "target": "current" },
  "display_text_locked": "Lock:即時揭露當前敵人弱點類型"
}
```

**lock_class: "required" — 試探斬的揭露 + 戰鬥**

```json
{
  "spec": { "action": "reveal_binary", "subject": "is_weakness_impact", "target": "current" },
  "display_text": "Lock 時揭露當前敵人弱點是否為衝擊(yes/no)"
}
```

## 九、玩家可見性對照

按呈現層原則:系統內部所有資料都精確,但呈現給玩家分層處理。

| 系統內部資料 | 玩家可見? | 預設緊湊 | 點擊展開 | 教學期額外 | 戰鬥紀錄 |
|------------|---------|---------|---------|---------|---------|
| `id` | ✗ | — | — | — | — |
| `name` | ✓ | 文字(卡面標題) | 同 | — | 同 |
| `resource_class` | ✓ | 圖示或邊框顏色 | 加註類別文字 | — | 同 |
| `function_class` | ✓ | 圖示或邊框顏色 | 加註類別文字 | — | 同 |
| `weakness_type` | ✓ | 圖示+色塊(顯眼) | 加註類型文字 | — | 同 |
| `contribution`(類型) | ✓ | 圖示(顯示是哪一類) | 同 | — | 同 |
| `contribution`(數字) | ✓ | 顯示「+N 衝擊」 | 同 | 同 | 同 |
| `strength_level` | ✓ | ★(醒目) | 同 | — | 同 |
| `card_form` | △ | (隱含於介面分區) | — | — | — |
| `pool_remaining`(池剩餘) | ✓ | 池視覺(填滿狀態) | 數字 | 同 | 同 |
| `remaining_uses`(個體) | ✓ | 點數圖填充(如 ●●○) | 同 + 數字 | 同 | 同 |
| `lock_class` | ✓ | 卡面標示(無 / 雙模式 / 🔒) | 同 | 同 | 同 |
| `special_effect.spec` | ✗ | — | — | — | 顯示為文字 |
| `special_effect.display_text` | ✓ | 圖示或關鍵字 | 完整文字 | 同 | 同 |
| `description` | ✓ | 隱藏(緊湊不顯示) | 完整顯示 | 同 | 同 |

**圖例**:✓ 玩家可見 / ✗ 玩家完全不可見 / △ 條件性可見

### 三類卡牌的卡面設計差異

**類 a:`"none"`** — 卡面**不寫任何 lock 相關文字**,純粹顯示:名稱、類別、貢獻、強度、使用次數。

**類 b:`"optional"`** — 卡面分為**兩段效果**:

```
[偵察] 情報卡 ★
使用次數: ●●

❒ 不 lock(預設):
  結束本擊時揭露此連戰所有
  剩餘敵人的弱點類型

❑ Lock 啟動:
  即時揭露當前敵人弱點類型
```

**類 c:`"required"`** — 卡面標**🔒 強制 Lock 標示**,寫清楚「使用時須 lock」+「flexible 卡需分配類型」:

```
[試探斬] 試探卡 ★ 🔒
+1 衝擊
使用次數: ●●●

🔒 Lock 啟動(必須):
  揭露當前敵人弱點是否
  為衝擊(yes/no)
```

## 十、卡牌獲得的隨機性

### 採用方式

每局獲得不同卡牌組合,但**玩家看見並選擇**,不做盲抽。

### 實作建議

- 整備期提供 3-4 張候選卡,玩家挑 1 張
- 卡片屬性變化時讓玩家看見完整資訊再選擇
- 不允許「隨機附加效果」這種失控變化

### 卡片變異的可能形態(類型計數模型下)

同樣的個體卡可能出現次數貢獻變化(例如某張爆發卡這次給 +3 衝擊,下次給 +2 衝擊 +1 穿刺),或同類卡牌池可能臨時增加類型張數(這局衝擊池 5 張、下局穿刺池 6 張)。讓玩家每局重新評估資源分布。

## 十一、多角色設計

### 角色不只是一張角色牌的差異

要讓多角色有意義,每個角色不能只差一張角色牌——他們應該有**整體玩法傾向的不同**。

每個角色實際上是「**一套不同類型偏好的資源池 + 數張個體角色卡**」。差異可從:
- 資源池中各類型張數比例
- 角色卡的特殊機制
- 爆發卡組合

三方面建立。

### 角色範例

- **狩獵者**:資源池偏穿刺與情報,角色卡擅長「揭露後一擊」,鼓勵先偵察再擊殺
- **狂戰士**:資源池偏衝擊與通用,角色卡擅長「無視類型修正」,鼓勵直接賭博
- **學者**:資源池偏複合卡,角色卡擅長「跨場連鎖揭露」,鼓勵情報累積

### 開發建議:漸進式內容擴張

**先做 1 個角色把核心打磨完,再考慮第二個角色作為第一個正式更新**——這是最健康的開發路徑。如果一開始就規劃 5 個角色,每個都會被攤薄。

對單人開發或小團隊尤其重要。

## 十二、設計新卡的檢查清單

新增一張卡前,必須通過以下所有檢查。

### 設計合約三問

1. **這張卡的「資源類別」是什麼?**(工具 / 爆發 / 角色)
2. **這張卡的「功能類別」是什麼?**(戰鬥 / 情報 / 複合)
3. **這張卡的「卡牌形態」是什麼?**(資源池 / 個體)

### Lock 機制檢查

4. **這張卡的 lock_class 是什麼?**(none / optional / required)
5. **如果是 optional,雙模式效果如何不同?**(廣度 vs 深度 的明確取捨)
6. **如果是 required,為什麼?**(揭露需要時間性?flexible 需要分配?)

### contribution 檢查

7. **這張卡的 contribution 是什麼?**(明確類型 / flexible / null)
8. **強度層級對應 contribution 嗎?**(★ = +1,★★ = +2,★★★ = +3)

### special_effect 檢查(若有)

9. **觸發時機與 lock_class 一致嗎?**
10. **action 是否在現有列表中?**(避免新增冗餘 action)
11. **display_text 是「人話」嗎?**(玩家能直接讀懂)

### 設計差異化檢查(對個體卡)

12. **這張卡與既有卡的「使用情境」是否真的不同?**——避免「兩張卡感覺一樣」
13. **這張卡能不能被另一張現有卡 + 其他資源組合替代?**——若是,設計不獨立
14. **這張卡的「招牌時刻」在哪場戰鬥?**——說不出來代表設計沒有焦點

### 與 OHK 美學的相容性檢查

15. **這張卡是否引入「機率性成敗」?**(必須 ❌)
16. **這張卡是否能「跨整備循環攜帶累積效益」?**(必須 ❌)
17. **這張卡是否讓玩家用「算術」而非「判斷」?**(必須 ❌)

通過所有檢查才能進入資料庫。
