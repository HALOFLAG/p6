## 二、結算機制快速摘要

> 完整機制見 [遊戲核心系統機制.md 第二章](遊戲核心系統機制.md)。

- 每個敵人有一張**需求表**:各類型需要多少「次數」+ 一個「混合任意需求」
- 玩家本擊每張卡貢獻一個或多個類型的次數(`contribution` 欄位)
- 結算 = 加總本擊各類型次數,**任一條件達標即 OHK**
- **無倍率、無詞條**
- 卡牌的 special_effect 觸發時機依 `lock_class` 而定

## 三、Layer A:資源池卡(14 張,全部 `lock_class: "none"`)

資源池卡是同質卡——同一個池內每張卡完全相同,玩家決策是「從這池投幾張使用」。所有資源池卡 `lock_class: "none"`,可直接放本擊區、結束本擊時統一結算。

### 衝擊池

| 卡 | id | ★ | base | use_limit | pool_size | 池總供給 | lock_class |
|----|----|---|------|-----------|-----------|--------|----|
| 直擊 | `tool_pool_impact_basic` | ★ | +1 衝擊 | 3 | 3 | 9 衝擊 | none |
| 重斬 | `tool_pool_impact_strong` | ★★ | +2 衝擊 | 2 | 2 | 8 衝擊 | none |

### 穿刺池

| 卡 | id | ★ | base | use_limit | pool_size | 池總供給 | lock_class |
|----|----|---|------|-----------|-----------|--------|----|
| 銳刺 | `tool_pool_pierce_basic` | ★ | +1 穿刺 | 3 | 3 | 9 穿刺 | none |
| 貫穿 | `tool_pool_pierce_strong` | ★★ | +2 穿刺 | 2 | 2 | 8 穿刺 | none |

### 燃燒池

| 卡 | id | ★ | base | use_limit | pool_size | 池總供給 | lock_class |
|----|----|---|------|-----------|-----------|--------|----|
| 點火 | `tool_pool_burn_basic` | ★ | +1 燃燒 | 3 | 2 | 6 燃燒 | none |
| 預判燒灼 | `tool_pool_burn_strong` | ★★ | +2 燃燒 | 2 | 2 | 8 燃燒 | none |

### Layer A 規格 JSON 範例

```json
// 直擊
{
  "id": "tool_pool_impact_basic",
  "name": "直擊",
  "resource_class": "tool",
  "function_class": "combat",
  "card_form": "pool",
  "weakness_type": "impact",
  "contribution": { "impact": 1 },
  "strength_level": 1,
  "pool_size": 3,
  "use_limit": 3,
  "lock_class": "none",
  "special_effect": null,
  "description": "對單一目標貢獻 +1 衝擊"
}
```

(其他資源池卡格式相同,只差 weakness_type / contribution / pool_size / use_limit)

**對應卡牌設計原則**:第二章(工具層)、第四章(資源池形態)、第五章(lock_class: none)、第六章(contribution)。

## 五、Layer B:試探卡(3 張,`lock_class: "required"`,只作用於當前敵人)

### 設計原則:試探卡是**動作**,只能作用於當前敵人;Binary 揭露用於「驗證猜測」

試探卡與情報卡的核心區分:
- **情報卡** = 「我完全不知道,想要答案」(揭露具體類型/數值/狀態)
- **試探卡** = 「我有猜測,想驗證」(binary yes/no 確認)

| 卡 | id | base | use_limit | Lock 揭露 |
|----|----|------|-----------|---------|
| 試探斬 | `compound_impact` | +1 衝擊 | 3 | 揭露當前敵人**弱點是否為衝擊**(yes/no) |
| 探刺 | `compound_pierce` | +1 穿刺 | 3 | 揭露當前敵人**最低需求是否 ≤ 2**(yes/no) |
| 試探擊 | `compound_generic` | +1 任一(flexible) | 3 | 揭露當前敵人**失敗是否會 GAME OVER**(yes/no) |

## 七、Layer B:角色卡(4 張)

| 卡 | id | 弱點類型 | base | use_limit | lock_class | 特殊效果 |
|----|----|---------|------|-----------|----|---------|

| 致命凝視 | `character_focus` | 穿刺 | +4 穿刺 | 1 | none | 對連戰最高需求者所有貢獻 +1 |
| 全知之眼 | `character_omniscient` | 無 | 0 | 1 | **required** | 揭露此連戰所有剩餘敵人的**完整資訊**(弱點+需求表+敵人類別+戰鬥狀態 instance) |
| 洞察一擊 | `character_insight` | 通用 | +2 任一(flexible) | 1 | **required** | Lock 揭露當前敵人弱點 + flexible 分配 |

### 全知之眼的差異化

> **全知之眼揭露的不只是「弱點+需求+敵人類別」**,還包含**每隻敵人的具體戰鬥狀態 instance**(這是 layer 3 的即時狀態,通常進入戰鬥才知道)。
> 
> 三張情報卡(不 lock)合用 ≈ 揭露剩餘敵人的弱點+需求+敵人類別(layer 4),但**不揭露戰鬥狀態 instance**。
> 
> 全知之眼是「招牌」——它揭露玩家本來要進入戰鬥才能知道的「先攻/被襲」變體,讓玩家可以提前規劃連戰流程。**這是不可替代的價值**。

## 九、設計驗算:對 5 個敵人模板的擊殺路徑

對應 [敵人資料庫.md](敵人資料庫.md) 的 5 個原型敵人,以新模型驗算可行性。**任一條件達標即 OHK**。

### 哨兵(衝擊 1-2 / 穿刺 4-5 / 混合 4-8)

| 實例 | 推薦本擊 | 計算 | 結果 |
|------|---------|------|------|
| 弱衝擊低(衝擊 1) | 直擊 ×1 | +1 衝擊 = 1 | ✓ 達標 |
| 弱衝擊中低(衝擊 2) | 直擊 ×2 或 重斬 ×1 | +2 衝擊 = 2 | ✓ 達標 |
| 弱穿刺低(穿刺 1) | 銳刺 ×1 | +1 穿刺 = 1 | ✓ 達標 |
| 弱穿刺中低(穿刺 2) | 銳刺 ×2 或 貫穿 ×1 | +2 穿刺 = 2 | ✓ 達標 |

### 巡守(弱點 2-4)

| 實例 | 推薦本擊 | 計算 | 結果 |
|------|---------|------|------|
| 弱衝擊高(衝擊 4) | 重斬 ×2 | +4 衝擊 = 4 | ✓ 達標 |
| 弱穿刺中高(穿刺 3) | 貫穿 ×1 + 銳刺 ×1 | +2+1 穿刺 = 3 | ✓ 達標 |
| 弱燃燒高(燃燒 4) | 預判燒灼 ×2 | +4 燃燒 = 4 | ✓ 達標 |

### 重甲衛兵(弱點 2-4)

| 實例 | 推薦本擊 | 計算 | 結果 |
|------|---------|------|------|
| 弱衝擊高(衝擊 4) | 重斬 ×2 | +4 衝擊 = 4 | ✓ 達標 |
| 弱穿刺中(穿刺 2) | 貫穿 ×1 | +2 穿刺 = 2 | ✓ 達標 |

### 伏擊者(弱點 1-2,被襲狀態下情報減半)

| 實例 | 推薦本擊 | 計算 | 結果 |
|------|---------|------|------|
| 弱穿刺低(穿刺 1) | 銳刺 ×1 | +1 穿刺 = 1 | ✓ 達標 |
| 弱燃燒中(燃燒 2) | 預判燒灼 ×1 | +2 燃燒 = 2 | ✓ 達標 |

### 領主(弱點 4-6,核心挑戰)

| 實例 | 推薦本擊 | 計算 | 結果 |
|------|---------|------|------|
| 弱衝擊高(衝擊 4) | 重擊 ×1 + 直擊 ×1 | +3+1 衝擊 = 4 | ✓ 達標 |
| 弱衝擊極高(衝擊 6) | 重擊 ×1 + 重斬 ×1 + 直擊 ×1 | +3+2+1 = 6 | ✓ 達標(3 張) |
| 弱穿刺高(穿刺 4) | 致命凝視 ×1 | +4 穿刺 = 4 | ✓ 達標(1 張單卡 OHK!) |
| 弱穿刺極高(穿刺 6) | 致命凝視 ×1 + 貫穿 ×1 | +4+2 = 6 | ✓ 達標(2 張) |

## 十、已知盲點與測試觀察

### 設計亮點

- **致命凝視單卡 OHK 領主穿刺**:這是「招牌動作儀式化」最強烈的演出時刻
- **殺戮意志的雙類型貢獻**:對 hedge 不下決心的玩家是「兩邊都過一半」的安全網,對讀懂弱點的玩家是「把多的浪費掉」的選擇
- **試探卡 +1 + binary 揭露**:精煉的「驗證猜測」工具,跟情報卡的「探索答案」清楚區分
- **情報卡 optional lock**:玩家在「現在 vs 未來」之間有真實的判斷空間

### 觀察點(待測試驗證)

1. **無本擊上限的影響**:玩家會不會出現「對最後敵人 dump 所有資源」的行為?(詳見 [本擊上限討論.md](本擊上限討論.md))
2. **致命凝視太強?**:對連戰最高需求者 +1 修飾 + 4 穿刺基礎 = 對領主穿刺幾乎是「神器」。是否需要平衡?
3. **flexible 機制的玩家理解度**:爆裂、洞察一擊 lock 時分配是否直覺?
4. **燃燒池只 14 供給**:對巡守+伏擊者都弱燃燒的極端情境(理論上 4+2=6 需求),供給足夠但餘裕較少
5. **Lock 機制的學習成本**:三類 lock 行為玩家能掌握嗎?
6. **情報 vs 試探的區分**:玩家會直觀理解「探索 vs 驗證」嗎?
