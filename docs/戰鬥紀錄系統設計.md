## 一、設計概念與定位

### 核心概念:**「我們的行動製造了地圖」**

不同於 Slay the Spire / FTL 等遊戲「**給你地圖、讓你選路**」的設計,P6 採用:

> **玩家的每場戰鬥、每次失敗、每個整備節點,都會在地圖上長出對應的標記。** 地圖不是預設,是玩家走過的痕跡。

這對應 [遊戲設計概念.md](遊戲設計概念.md) 的:
- **理解即進度**(地圖是元知識累積的物理紀錄)
- **失敗有有機後果**(失敗在地圖上留下疤痕)
- **紀錄是讓玩家看見自己成長的鏡子**(§1 三條流的紀錄流)

### 框架:冒險手記 vs 資料庫

戰鬥紀錄**不是查資料庫**,是**翻冒險日誌**:

| 框架差異 | 資料庫式 | 冒險手記式(P6 採用) |
|---------|--------|----------------|
| 玩家代入感 | 工具性查詢 | 主角寫日記 |
| 視覺風格 | 表格 / 列表 | 手繪羊皮紙 + 像素插畫 |
| 失敗痕跡 | 「失敗 1 次」數值 | X 標記 + 黑色虛線(視覺) |
| 跨戰役 | 篩選器選戰役 | 翻書頁 / 大羊皮紙地圖 |
| 「精彩時刻」 | 統計欄位 | 翻到那一頁的記憶 |

→ **冒險手記框架**讓「回顧」變成有敘事重量的動作,而非工具性查詢。

### 主要解決的問題

| # | 問題 | 解法 |
|---|------|------|
| 1 | 精英對戰前的「記憶調出」 | 玩家從地圖點 X / 紅色虛線 → 翻到對應頁面 |
| 2 | 校準對話的精確版承載地 | 校準對話「詳細數據」連結 → 翻到該場戰鬥頁面 |
| 3 | 失敗痕跡的視覺化 | X 標記 + 黑色虛線(永久保留在地圖) |
| 4 | 元知識累積的外部備忘 | 玩家可主動翻看過去戰鬥 |
| 5 | 跨戰役的成就感 | 大羊皮紙地圖長期累積 |

## 二、核心結構

戰鬥紀錄系統有**兩個主視圖**:

### 2.1 地圖視圖(總覽)

一張**大羊皮紙地圖**,跨戰役持續累積。主冒險路線是**一條長線**,各戰役是路上的**站點**。

```
           ❓
            \\\\
●━━━●━━━●━━━━●━━━●━━━━━○━━━━━○
 戰役 1                  戰役 2  戰役 3
                         (未走)  (未走)
       ●━━━━━●
              \\\\X
               (失敗痕跡,黑色虛線退回)
```

點擊地圖上任一節點 → **翻到該節點的頁面視圖**。

### 2.2 頁面視圖(詳細)

一本**冒險手記**,每個節點對應一張**雙頁書頁**:
- 左頁 + 右頁 構成一個節點的完整紀錄
- 翻書頁 = 切換到上一個 / 下一個節點
- 節點類型不同,雙頁內容不同(詳見 §4)

## 三、地圖規格

### 3.1 節點類型

| 節點 | 視覺 | 對應 |
|------|------|------|
| **戰鬥節點(成功)** | 紅色實心點 | 已完成的戰鬥 |
| **戰鬥節點(失敗)** | X 標記 | 玩家被精英擊敗處 |
| **整備節點** | 待設計(可能營火 / 帳篷圖示) | 連戰之間的整備 |
| **戰役入口(已走)** | 已連入主線的站點 | 完成過的戰役 |
| **戰役入口(未走)** | 空心圈 | 未來戰役 |
| **追擊路徑端點** | 問號 | 敵人逃跑方向,玩家可選擇追擊 |

### 3.2 連線類型

| 連線 | 視覺 | 含義 |
|------|------|------|
| 已走過 | 黑色實線 | 玩家正常前進的路徑 |
| 敵人逃跑方向 | **紅色虛線** | 兔/狼型失敗後敵人逃離的方向;狼型會在線末變精英 |
| 玩家退回方向 | **黑色虛線** | 玩家在精英節點失敗後,退回上個整備節點 |
| 未走主線 | 淡色實線(可選) | 主冒險的未來路徑(視覺暗示) |

### 3.3 跨戰役地圖

**一張大羊皮紙**承載所有戰役:
- 主冒險路線是**主軸長線**
- 各戰役是線上的**站點**(以章節標題標示:序章 / 戰役 2 / 戰役 3...)
- 站點之間有過場連線(可能是「旅程」「日記跳頁」等視覺表現)
- 玩家點擊戰役站點 → **進入該戰役頁面(地圖視圖局部放大)**
- 玩家點擊個別節點 → 翻到該節點的雙頁書頁

### 3.4 「玩家行動製造地圖」的具體運作

| 玩家行動 | 地圖變化 |
|---------|--------|
| 完成第 1 場戰鬥 | 戰役頁出現第 1 個紅色實心點 |
| 完成整備節點 | 出現整備節點圖示 + 連線 |
| 失敗一隻兔子 | 紅色虛線從該點延伸到末端(敵人逃離,但不變精英) |
| 失敗一隻狼(第 1 次) | 紅色虛線延伸到連戰末段,**末段出現新精英節點** |
| 失敗一隻狼(第 2+ 次)| 紅色虛線延伸到「**未解決池**」(地圖邊緣?問號?) |
| 在精英節點失敗 | 該節點變 X 標記 + 黑色虛線退回上個整備節點 |
| 玩家追擊失敗的狼 | 從整備節點沿紅色虛線前進到精英節點 |
| 完成整場戰役 | 該戰役站點變「已完成」標記 |
| 完成全主冒險 | 整張地圖填滿(可能是結局演出) |

## 四、頁面規格

### 4.1 戰鬥節點頁面(成功 / 失敗)

```
┌─────────────────────────┬─────────────────────────┐
│  左頁:使用的卡           │  右頁:敵人              │
│                          │                          │
│  ┌───┬───┬───┬───┬──┐  │  ┌─────────────┐         │
│  │卡 1│卡 2│卡 3│卡 4│..│  │  [敵人像素圖]│         │
│  └───┴───┴───┴───┴──┘  │  │  (頭頂標記)  │         │
│  (該場全部 commit 卡,    │  └─────────────┘         │
│   縮略圖,點擊展開詳細)   │                          │
│                          │  [敵人名稱 / 類別 / 標籤] │
│  ─────────────────────   │  ─────────────────────   │
│                          │                          │
│  (文字資訊欄)            │  (文字資訊欄)            │
│                          │                          │
└─────────────────────────┴─────────────────────────┘
```

#### 左頁:使用的卡

- **顯示該場戰鬥打出的全部卡**(P6 一次戰鬥只 commit 一次,所以是該本擊的完整卡組)
- 卡牌縮略圖**排列在書頁上**(像貼紙 / 書頁裝飾)
- **點擊縮略圖 = 展開該卡的完整詳細介面**
- flexible 卡顯示玩家當時的分配結果

#### 右頁:敵人

- **敵人圖**(像素風插畫)+ 頭頂標記(具體含義尚未設計,可能是名字 / 類別 / 戰鬥狀態)
- **該戰鬥中已揭露的敵人資訊**(動態,反映玩家當時知道的):
  - 若玩家用了估量 / 偵察 / 警示 → 顯示揭露的內容
  - 若玩家沒投資情報 → 只有層 1/2 可見資訊
- **戰鬥結果類型**(對應 6 對話狀態之一):
  - 大幅 Overkill / 精準擊殺 / 勉強過 / 差類型錯 / 差量不足 / 大幅不足

#### 雙頁底部文字欄

- 預留給敘事 / 校準對話精確版 / 玩家心境描述

### 4.2 整備節點頁面

```
┌─────────────────────────┬─────────────────────────┐
│  左頁:劇情 / 對話       │  右頁:地點美術圖        │
│                          │                          │
│  (手寫風格的文字記錄)    │  ┌─────────────────┐    │
│                          │  │  [地點美術圖]   │    │
│  例:                     │  │  (羊皮紙風景)   │    │
│  「克勒斯靠在火堆邊       │  └─────────────────┘    │
│   翻看背袋,柯登          │                          │
│   清點剩餘藥材。」        │  [地點名稱]              │
│                          │                          │
│  [NPC 對話片段]          │  (卡牌回補資訊?)         │
│                          │                          │
└─────────────────────────┴─────────────────────────┘
```

- **左頁**:手寫風格的劇情 / 對話概要
- **右頁**:該整備節點對應地點的美術圖
- 翻書頁 = 上 / 下一個節點
- 具體還有什麼元素(卡牌回補資訊 / 紀念品 / 玩家自寫筆記)待後續細化

### 4.3 戰役入口頁面(待設計)

點擊地圖上「空心圈」未來戰役入口時,可能呈現:
- 該戰役的標題 / 章節名
- 解鎖前提示(若有)
- 進入戰役的按鈕

第一期只有序章,此頁面內容簡單。

## 五、互動設計

### 5.1 查閱時機

**任何非劇情演出時刻都可查看戰鬥紀錄**:
- Hub 內
- 整備節點
- 戰役選擇介面
- 戰鬥前
- 戰鬥結束後的校準對話畫面

**唯一例外**:重要劇情段播放中、戰鬥的 Place/Lock 階段(避免被當作即時試算表)。

### 5.2 入口按鈕

| 觸發位置 | 按鈕視覺 | 行為 |
|---------|--------|------|
| 全局 UI 角落 | 書本 / 地圖圖示 | 開啟手記(預設翻到當前進度) |
| 精英敵人對戰前 | 特別強調的按鈕 | 自動翻到「**上次面對該敵人**」的頁面 |
| 校準對話「詳細數據」連結 | 文字連結 | 翻到該場戰鬥頁面 |

### 5.3 導航方式

| 方式 | 行為 |
|------|------|
| 翻書頁 | 上一個 / 下一個節點(按時間順序) |
| 點地圖節點 | 跳到該節點的雙頁書頁 |
| 索引(可選,暫不做) | 按敵人類別 / 戰役 / 結果類型篩選 |

**敵人類別索引**:**第一期暫不做**,等有實際需求(例如戰役 2 開始後資料量大)再加。

### 5.4 精英對戰前的具體流程

```
精英敵人即將開戰 
   → UI 顯示「📖 翻閱手記」按鈕(特別強調,明確告知這敵人是怎麼來的)
   → 玩家點擊 
   → 自動翻到該敵人「上次戰鬥」的頁面
   → 玩家看完(可以看左頁卡牌 / 右頁結果 / 文字欄)
   → 點「返回戰鬥」按鈕 
   → 進入精英戰
```

獵人模式 / 騎兵模式都採此流程,差別在「**按鈕視覺強調程度**」(獵人模式更醒目,騎兵模式較簡)。

### 5.5 卡牌詳細檢視

- 縮略圖排在書頁上 = 預設緊湊呈現
- 點擊縮略圖 = 展開該卡的完整資訊(類型、貢獻、特殊效果等)
- 展開後可關閉回到頁面

## 六、資料結構需求

### 6.1 戰鬥節點紀錄

```
BattleRecord {
  battle_id: String
  campaign_id: String
  connection_position: int  // 在連戰中的位置
  retry_count: int          // 是否為重新挑戰(0 = 第一次)
  
  enemy: {
    template_id: String
    instance_id: String
    combat_state: String     // 先攻/標準/被襲
    is_elite: bool
    is_elite_from_failure: bool  // 是否為玩家失敗造成的精英
  }
  
  revealed_info: {           // 該場戰鬥中,玩家當時已揭露的資訊
    weakness_type: String?
    full_requirements: Dictionary?
    enemy_class: String?
    used_intel_cards: Array
    used_probe_cards: Array
  }
  
  player_strike: {
    cards: Array<{card_id, flexible_allocation?}>
    contributions: Dictionary  // 各類型實際次數
  }
  
  result: {
    type: String  // 6 對話狀態之一
    success: bool
    requirement_table: Dictionary  // 該敵人的完整需求
    contribution_table: Dictionary  // 玩家本擊各類型總和
  }
  
  timestamp: int  // 用於排序
}
```

### 6.2 整備節點紀錄

```
PrepNodeRecord {
  node_id: String
  campaign_id: String
  connection_position: int
  
  scene_id: String  // 對應地點美術圖
  narrative_segment_id: String  // 對應劇情 / 對話片段
  
  card_pool_state_before: Dictionary
  card_pool_state_after: Dictionary  // 整備期回補後
  
  timestamp: int
}
```

### 6.3 地圖資料

```
CampaignMap {
  campaign_id: String
  nodes: Array<{node_id, position, type}>
  connections: Array<{from_id, to_id, type}>  // type: solid / red_dashed / black_dashed / faint
  
  // 失敗痕跡
  failure_points: Array<{node_id, retry_count, escape_direction}>
}
```

### 6.4 跨戰役資料

```
AdventureRecord {
  campaigns_completed: Array<campaign_id>
  current_campaign: campaign_id?
  
  battle_records_by_enemy: Dictionary<enemy_template_id, Array<battle_id>>  // 索引用
  battle_records_by_campaign: Dictionary<campaign_id, Array<battle_id>>
  
  // 為精英對戰前的「上次紀錄」查詢用
  last_failure_by_enemy: Dictionary<enemy_template_id, battle_id>
}
```

### 6.5 儲存位置

- 主資料:`user://battle_records.json`(玩家資料夾)
- 跟存檔系統獨立(存檔回滾不影響紀錄)
- **失敗紀錄永久保留**,即使重新挑戰成功也不抹除(保留疤痕)

## 七、與其他系統的整合

### 7.1 校準對話系統

- 校準對話只給「方向感」(對應 [遊戲核心系統機制.md] §10)
- 對話介面提供「**詳細數據**」連結
- 點擊 → 翻到該場戰鬥的雙頁書頁

### 7.2 精英機制(失敗代價系統)

- 狼型失敗 = 變精英排末 → 地圖紅色虛線延伸
- 精英對戰前 = 自動提示「翻閱手記」按鈕
- 點擊 = 翻到上次紀錄

### 7.3 失敗重啟機制(資源持續縮減模型)

- 失敗 → X 標記出現在地圖
- 玩家選「重新挑戰」 → 黑色虛線退回上個整備節點
- **紀錄永久保留**,新挑戰會生成新節點(retry_count + 1)
- 完美通關判定:檢查整場戰役有無 X 標記

### 7.4 紀念品系統

- **待後續設計**——可能方向:
  - 紀念品就是手記中「夾頁物件」(乾燥花 / 信件 / 票根)
  - 翻到特定頁面時自然顯示
  - 仍遵守「不顯示未解鎖位」原則(沒解鎖的紀念品不在頁面出現)

### 7.5 Hub

- **整合方式待 UI 階段討論**
- 可能方向:Hub 中央就是攤開的手記;地圖 = Hub 的主要視覺
- 也可能 Hub 跟手記分開,手記從 Hub 一個按鈕進入

## 八、美術風格暫定方向

| 元素 | 風格 |
|------|------|
| 手記書頁、地圖、UI 邊框 | **手繪 / 羊皮紙風** |
| 卡牌縮略圖、敵人插畫、玩家角色 | **像素風** |
| 文字資訊欄 | 手寫字體(或印刷風字體模擬手寫) |
| 地圖標記(節點 / 連線) | 紅 + 黑 為主的手繪標記 |

整體感:**冒險日誌**——像被翻舊的探險家手記,夾雜玩家自繪的小圖。

**注意**:這是**暫定方向**,可能調整(對應 [UI 設計指引.md](UI%20設計指引.md) §9)。
