## 一、文件用途與符號規範

### 用途

- **粗略骨架階段**:只方框 + 元件名稱,不含尺寸、不含具體視覺風格
- **方向確認用**:讓設計者(使用者)能快速確認佈局,不投入美術細節
- **實作參考**:給 M0-M4 階段的 AI 輔助寫 code 一個視覺基準

### ASCII 符號規範

| 符號 | 含義 |
|------|------|
| `┌─┐ └─┘` | 一般方框(普通元件 / 卡牌縮略圖) |
| `┏━┓ ┗━┛` | 強調方框(Lock 卡定格、重要 UI 元素) |
| `╔═╗ ╚═╝` | 牌堆 / 庫存區方框 |
| `╭─╮ ╰─╯` | 圓角方框(對話泡、彈出視窗) |
| `═══` | 舞台地面 / 區塊分隔線 |
| `▓` | 模糊剪影 / 灰階區塊 |
| `███` | 黑色剪影 / 立繪佔位 |
| `●●▶○○` | 進度條(已完成/當前/未完成) |
| `🔒` | Lock 強制標示 |
| `★` | 強調 / 揭露標記 |

## 二、戰鬥介面 — 版型來源與設計原則對應

### 版型來源

戰鬥介面版型參考兩款遊戲:

1. **One Turn Kill(OTK)** — 側面舞台式雙角色同框(玩家左 / 敵人右),底部手牌列
2. **使用者提供的 mockup** — 雙頭像(玩家+NPC)左上、多敵人剪影、敵人情報面板右側、本擊區時間線、手牌扇形+多牌堆

### 與 OTK / 寶可夢式的對比

| 面向 | Pokémon 式 | **OTK 式(P6 採用)** |
|------|---------|------------------|
| 玩家角色 | 不可見(玩家=攝影機) | **可見立繪**,在左側 |
| 敵人位置 | 畫面中央偏上 | 右側,跟玩家同高 |
| 視角 | 從玩家後方看 | 側面平視,雙方同框 |
| 場景 | 抽象背景 | **具體舞台**(連戰地點) |
| 戲劇感 | 弱 | 強(觀眾看擂台) |
| 角色 identity | 弱 | 強(玩家是個明確角色) |

**為什麼選 OTK 式**:
- 敵人視覺線索是 P6 核心遊玩 → 側面全身比正面更能看持械/鎧甲/姿態
- 四類敵人(兔/狼/豹/熊)是物種辨識 → 全身像比剪影差異更大
- 多角色系統(未來)→ 玩家角色有 identity
- 「騎兵斬殺」隱喻 → 玩家是行動者
- 儀式化擊殺 → 側面舞台讓擊殺瞬間有演出感

### 設計需求對應

| 設計原則 / 規範 | 此版型如何符合 | 來源 |
|--------------|------------|------|
| 對稱即誠實 | 玩家立繪 + 敵人立繪同框,雙方視覺平等 | 設計概念 §3 原則二 |
| 視覺線索可讀(層 2 老手讀眼) | 敵人立繪提供持械/鎧甲/姿態判讀空間 | 敵人設計原則 §4 |
| 四層資訊架構 | 敵人情報欄預設只顯示層 1;層 4 顯示「???」未揭露;情報卡後揭露 | 核心系統機制 §3 |
| Place / Lock 兩態 | 時間線上 Place(P 標記)與 Lock(粗框+定格)視覺區分 | 核心系統機制 §1 |
| 玩家是判斷者,系統是計算者 | 本擊狀態即時加總(玩家不用算);敵人情報只呈現客觀資訊不推薦 | 核心系統機制 §10 |
| A 類 UI(整理不判斷) | 牌堆切換、扇形展開都是玩家自選工具 | UI 設計指引 §1 |
| 緊湊呈現 + 點擊展開 | 卡牌縮略圖、未揭露用 `???` 摺疊、剪影點擊才展開 | UI 設計指引 §1 |
| 雙聲道對話載體 | 玩家 + NPC 雙頭像左上,對應觀察視角 + 決策視角 | 核心系統機制 §8 |
| 連戰進度可感知 | 頂部 `●●▶○○` 進度條 | 戰役結構與節奏設計 §2 |
| 無 HP / 無養成 | 無玩家血量條、無經驗值、無等級顯示 | 設計概念 §6 |
| 多角色預留 | 玩家立繪位置可換不同角色 | 卡牌設計原則 §11 |
| 戰鬥紀錄任意時刻可查 | `📖手記` 按鈕在頂部常駐 | 戰鬥紀錄系統設計 §5 |

## 四、戰鬥介面 — 區塊說明

### 4.1 頂部 — 雙頭像 + 選單列 + 連戰進度

- **左上角雙頭像**:玩家頭像(上)+ NPC 頭像(下),固定常駐
- **頭像狀態**:
  - **沉默**:半透明 / 暗淡
  - **發言**:亮起 + 對話泡浮出
  - 對應雙聲道對話的「**誰在說話**」視覺
- **選單入口**:`[選單]`(暫停/設置)+ `[📖手記]`(戰鬥紀錄)
- **連戰進度**:`●●▶○○`(已戰/當前/未戰),不展開為剪影

### 4.2 中段舞台 — 玩家立繪 + 敵人立繪 + 後續敵人剪影

- **玩家立繪**(左)+ **敵人立繪**(右):側面對視
- **舞台地面**(`═══`):視覺基準線
- **後續敵人模糊剪影**(右側):
  - 預設只看到「**有什麼存在**」
  - **點擊展開** → 浮出「層 1 模板資訊」(類別、弱點範圍、需求層級)
  - **不洩露** layer 3 即時狀態 / layer 4 隱藏

### 4.3 敵人情報欄(右側)— 客觀資訊呈現

**只顯示客觀資訊,不顯示交互結果**:

✓ 顯示:
- 模板名稱
- 敵人類別(兔/狼/豹/熊)
- 弱點範圍
- 需求層級(層 1 模糊)
- 揭露後的精確需求數值(層 4)

✗ 不顯示:
- 玩家當前本擊的達標狀態(屬於交互結果)
- 推薦的卡 / 卡片組合建議(B 類 UI 禁忌)
- 預測勝率 / 機率

### 4.4 本擊區(時間線)

- 卡牌**水平排列**
- Place 卡:正常框,標 `P`,可拖回庫存
- Lock 卡:粗框 + 底色定格(`┏━━┓`),不可移動,標 `🔒`(若為 required)
- 時間線**只代表「卡進入順序」,不代表「結算順序」**(結算是整體加總)

### 4.5 本擊狀態 + 結束本擊按鈕(右側)

**本擊狀態**:
- 顯示**各類型加總**(衝擊/穿刺/燃燒/混合)
- **不顯示「✓ 達標」標記**(避免洩露未揭露需求)
- 等到「結束本擊」後在**結算演出**才揭曉

**結束本擊按鈕**:
- 預設樣式 = 「→」(中性,不暗示成敗)
- 若有 required 卡未 lock = 禁用 + 提示

### 4.6 庫存區

**結構**(細節待「**特殊卡牌堆**」討論定案):
- 多個牌堆(衝擊池 / 穿刺池 / 燃燒池 / 特殊卡)
- 中間手牌扇形展示當前展開的池
- 點牌堆切換扇形內容

**互動**:
- 拖卡到本擊區 = Place
- 點卡 = 展開詳細卡面
- 點牌堆 = 切換扇形

## 五、戰鬥介面 — 風險決議記錄

從討論中產生的設計風險與決議:

| # | 風險 | 決議 | 狀態 |
|---|------|------|------|
| 1 | 雙模式 UI 差異(獵人 vs 騎兵) | 大致 UI 相同,只是部分對話/資訊隱藏。**待實機體驗驗證** | 🟡 延後 |
| 2 | 「✓ 達標」標記可能洩露未揭露需求 | **敵人資訊欄不顯示交互結果,只顯示客觀資訊**;達標揭曉移到「結算演出」 | ✓ 已採用 |
| 3 | 「特殊卡」牌堆是雜燴(情報+試探+爆發+角色) | **將開專題討論** | ⏳ 待討論 |
| 4 | 玩家立繪 vs 敵人立繪的視覺權重 | **待 UI 階段**(實際美術製作時調整) | 🟡 延後 |
| 5 | 多敵人 / 連戰預告 vs 層 3 規則(進戰才知道) | **採模糊剪影在右側顯示,點擊展開層 1 資訊** | ✓ 已採用 |

### 額外設計提醒(實作時不可違反)

| 提醒 | 對應原則 |
|------|--------|
| 本擊區時間線**不暗示順序影響結算**(結算是整體加總,Lock 順序只影響資訊獲取順序) | 類型計數結算模型 |
| 結算狀態**只顯示加總**,不分張累計 | 類型計數結算模型 |
| 「結束本擊」按鈕**不顯示達標預測** | 玩家是判斷者 |
| **沒有 HP 條 / 等級 / 經驗值 / 機率顯示** | OHK 設計拒絕清單 |
