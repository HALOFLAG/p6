## 一、里程碑哲學

### 核心原則

1. **增量交付**:每個 milestone 結束都**可玩、可驗證**——不是 "feature 50% 完成",是 "feature 100% 完成 + 整合可玩"
2. **機制優先 / 美術後置**:對應「**機制驗證可行後才投入美術資源**」原則,美術整合放在 M6(後期)
3. **每個 milestone 內含驗證閘**:**通過閘才算完成**,不通過就不繼續下一個
4. **風險前置**:把高風險的設計驗證(本擊上限、混合需求公式、雙模式 UI)放在前期 milestone

### 個人開發者的彈性

- **無時程壓力**:每個 milestone 標「**完整度關卡**」,不綁日期
- **預估顆粒度**:每個 milestone 標**迭代次數**(1 次迭代 ≈ 1-2 週,看當週時間)
- 若某個 milestone 卡住,可回頭調整設計(不是強推交付)

### 與第一期 Scope 的對應

| 第一期 Scope 必做項 | 落在哪個 milestone |
|------------------|------------------|
| 戰鬥系統 | M0-M1 |
| 卡牌資料 + 5 敵人 | M1 |
| 戰役框架 | M2 |
| 失敗代價系統 + 資源縮減模型 | M2 |
| 對話 + Hub | M3 |
| 教學 + 雙模式 UI | M4 |
| 戰鬥紀錄 + 紀念品 | M5 |
| 美術整合 | M6 |
| 音效 + 動畫 | M7 |
| 整體驗證 | M8 |

## 二、M0:技術 setup + 純機制 prototype + 舊專案評估

**目標**:用最簡的方式證明「**Place/Lock + 類型計數**」可以在 Godot 中運作 + 評估舊專案參考價值。

**預估迭代**:1 次(1-2 週)

### 範圍

- Godot 4.x 專案建立
- Git 設定 + .gitignore
- 1 張卡(直擊)+ 1 隻敵人(哨兵的 1 個 instance)
- 純文字 UI(Godot 預設 Label / Button)
- Place 卡 → 結束本擊 → 顯示結算結果
- **舊專案參考評估**:閱讀 `old_game_plan/` 內 Tier 1 的 3-5 個重點檔案(見 [程式規格書.md](程式規格書.md) §9),標記「實作時必看」清單

### 交付物

- 可執行的 Godot 專案(`.godot/` 可建)
- 一個能跑的「文字版戰鬥」:玩家按鈕 Place 卡,按結束本擊,顯示「OHK 成功 / 失敗」
- **舊專案重點檔案標記清單**(哪些值得後續 milestone 參考)

### 驗證閘

- [ ] 能用純文字介面 commit 一次本擊並看到結算結果
- [ ] 程式碼 push 到 Git
- [ ] Godot 在 Windows 環境穩定執行
- [ ] **3-5 個舊專案重點檔案讀過,標記出實作時要參考的部分**

### 為什麼這個 milestone 存在

- **驗證技術選型**:確認 Godot 可承載 P6
- **建立基礎架構**:後續所有模組接到這個骨架
- **AI 輔助寫 code 的熟悉**:第一次完整流程的試水
- **預先了解舊專案參考價值**:後續 milestone 遇到 UI / 拖放 / 主題等問題時可快速反查

### 建議閱讀的舊專案檔案(M0 期間)

| 檔案 | 為什麼讀 |
|------|--------|
| `old_game_plan/scripts/Ui/CardView.cs` | 卡牌 widget 結構 + 拖曳啟動模式 |
| `old_game_plan/scripts/Ui/PlayCardZone.cs` | drop zone 的 `_CanDropData` / `_DropData` 流程 |
| `old_game_plan/scripts/EventBus.cs` | 跨模組信號中繼的最簡實現 |
| `old_game_plan/scripts/Theme/UiTheme.cs` + `Colors.cs` | Theme 程式化建構 |
| `old_game_plan/scenes/main.tscn` | 場景樹組織模式參考 |

## 三、M1:戰鬥核心

**目標**:**完整單場戰鬥**可玩——所有卡、所有敵人、Place/Lock 完整實作。

**預估迭代**:2-3 次

### 範圍

- 28 張卡全部資料化(Godot Resource)
- 5 隻敵人模板資料化(每模板至少 1 instance)
- Place/Lock UI 雛形(基礎 Godot 元件,佔位視覺)
- 三類 lock_class 行為完整實作(none / optional / required)
- flexible 分配介面
- 類型計數結算 + OHK 判定
- 戰鬥狀態系統(先攻 / 標準 / 被襲)

### 交付物

- 一個能用佔位 UI 完成的「完整單場戰鬥」流程
- 可挑選任一敵人 + 任一卡組合測試
- 結算結果正確顯示(達標 / 未達)

### 驗證閘

- [ ] 5 隻敵人都能至少用 2 種方式擊殺(弱點路徑 + 混合路徑)
- [ ] 28 張卡都能在戰鬥中使用,行為符合設計
- [ ] Place 可拖、Lock 不可撤回
- [ ] flexible 卡可分配
- [ ] 情報卡 optional lock 兩種模式都生效
- [ ] **第三次紙上測試完成**,混合需求公式 / 本擊上限有結論

### 紙上測試任務(在此 milestone 完成)

- 測試混合需求公式(×4 vs ×1.7 of sum)→ 決定採用哪個
- 測試本擊上限(無上限 vs 動態上限)→ 決定是否引入

## 四、M2:戰役框架

**目標**:**完整單場戰役**可玩——連戰 + 整備 + 失敗代價 + 重啟邏輯。

**預估迭代**:2-3 次

### 範圍

- 連戰結構(序章 6 場戰鬥的編排)
- 整備節點介面(卡牌回補 + 自動存檔)
- 失敗代價依類別差異化(兔逃 / 狼精英化 / 豹熊 GO)
- 追擊機制(未解決池)
- 資源持續縮減失敗重啟模型
- 戰役狀態機(從第一場 → 整備 → 第二場 → ...)

### 交付物

- 一場完整序章戰役可從頭打到尾
- 失敗時有「重新挑戰 / 接受失敗」選擇
- 整備期回補正確(60-70% 上輪消耗)

### 驗證閘

- [ ] 能完整跑完一場戰役(無失敗版本)
- [ ] 能完整跑完一場戰役(含 1-2 次失敗 + 重新挑戰)
- [ ] 狼型精英排末機制正確運作
- [ ] 追擊階段運作
- [ ] 失敗 + 接受 → 回 Hub(雛形)正確
- [ ] 卡組消耗紀錄正確保留

## 五、M3:敘事 + Hub

**目標**:**完整敘事體驗**——重要劇情 + 道中對話 + 戰後校準 + Hub。

**預估迭代**:3-4 次

### 範圍

- 戰役開場 / 末尾劇情播放器(純文字版,Modal)
- 道中對話穿插(異步飄過字幕)
- Hub 雛形:基本場景 + NPC 對話 + 戰役選擇
- 戰後校準雙聲道對話 + 6 對話狀態
- 異步呈現機制(OHK 成功時不暫停遊戲)

### 前置決定(此 milestone 開始前必須決定)

- **NPC 性格定型**(沉默同行者 / 嚴肅長者 / 同伴熟靈 / ...)
- **序章主角身份**(從 [開頭劇情提案.md](開頭劇情提案.md) 17 候選方向選定)
- **序章具體劇情大綱**

### 交付物

- 從 Hub 進序章 → 完整劇情體驗 → 回 Hub
- 約 40-50 句的戰後校準對話
- 約 50-60 句的劇情段對話
- Hub 中 NPC 跟玩家對話

### 驗證閘

- [ ] 戰役開場 / 末尾劇情可播放
- [ ] 道中對話異步呈現不打斷戰鬥
- [ ] 6 種對話狀態能正確觸發
- [ ] 失敗時的 modal 對話強制停下,OHK 時的字幕異步呈現
- [ ] Hub 可重複進入,NPC 對話會變化

### 驗證閘

## 九、M7:音效 + 動畫精煉

**目標**:擊殺瞬間有「**儀式化**」感受。

**預估迭代**:1-2 次

### 範圍

- OHK 擊殺動畫(視覺強化 + 慢動作 + 特殊效果)
- 結算 bar 填充動畫(逼近達標線時的緊張感)
- 對話音效(打字音 / 確認音)

### 交付物

- 完整的音效 + 動畫包
- 擊殺瞬間有明確的儀式感

### 驗證閘

- [ ] 玩家在 OHK 成功時有「爽感」回饋
- [ ] 失敗時有明確的「壓抑」感受
- [ ] 不同敵人類別的失敗音不同
- [ ] 動畫速度可調(對應雙模式 UI)

## 十、M8:Alpha 測試

**目標**:**內部完整跑測 + 至少 3 位玩家測試**,收集反饋。

**預估迭代**:2-4 次(視反饋複雜度)

### 範圍

- 完整內部跑測序章戰役(獵人模式 + 騎兵模式)
- 邀請 3+ 位外部測試者
- 收集:通關率 / 體驗反饋 / Bug 清單 / 平衡問題
- 依反饋調整(可能回到前期 milestone 微調)

### 驗證標準(對應 [第一期開發Scope.md](第一期開發Scope.md) §7)

#### 機制驗證
- [ ] 5 隻敵人都有至少 2 條擊殺路徑
- [ ] 28 張卡都有合理使用情境
- [ ] Place/Lock 機制玩家可直觀理解
- [ ] 失敗代價依類別差異化能感受到
- [ ] 資源縮減模型在 2-3 次失敗後仍可遊玩

#### 節奏驗證
- [ ] 獵人模式完整跑完 25-30 分鐘,節奏舒適
- [ ] 騎兵模式完整跑完 8-15 分鐘,流暢無打斷
- [ ] 道中對話不打斷戰鬥節奏
- [ ] 戰後校準對話異步播放正確

#### 體驗驗證
- [ ] **至少 3 位玩家測試後通關率 > 60%**
- [ ] 玩家失敗時能直覺理解原因
- [ ] 雙模式 UI 切換流暢

#### 敘事驗證
- [ ] 序章作為自包含短篇有完整起承轉合
- [ ] NPC 對話量級適中(40-50 句不嫌煩)
- [ ] 紀念品事後呈現,玩家不會「為收集而玩」

## 十三、風險與緩解

| 風險 | 緩解策略 | 必須在哪個 milestone 處理 |
|------|--------|---------------------|
| 混合需求公式平衡不對 | M1 紙上測試,M2 結束前完成決定 | M1 結束 |
| 本擊上限有 dump 問題 | M1 結束時依實機體驗決定是否加 | M1 結束 |
| 美術風格決定卡住 | M6 開始前必須決定;若拖延,M6 延後 | M5 結束 |
| NPC 性格定型卡住 | M3 開始前必須決定 | M2 結束 |
| 序章主角身份未選 | M3 開始前必須選 | M2 結束 |
| 雙模式 UI 切換太複雜 | M1-M2 設計時預留架構,M4 整合 | M4 |
| 玩家無法理解 Place/Lock | M4 教學期 UI 強化 | M4 |
| 失敗代價系統感覺太重 | M2 測試時可微調 | M2 |
| 戰役 30 分鐘太長 / 太短 | M3 完整跑測時可調 | M3 |
| AI 輔助寫 code 效率不如預期 | 每個 milestone 結束評估,必要時簡化範圍 | 持續 |

### 整體風險:**範圍蔓延**

個人開發者最大的風險是「**功能蔓延**」——一直加東西,永遠做不完。緩解:

- **嚴格遵守 [第一期開發Scope.md](第一期開發Scope.md)** 的「不在 Scope」清單
- 任何想加的功能,先寫進「後續期數」,不混入第一期
- 每個 milestone 結束評估:「**這個 milestone 真的完成了嗎,還是我又加了東西?**」
